### 进程的概念、组成、特征

#### 进程的概念

程序：静态的，是存放在磁盘里的可执行文件，是指令的集合 。
进程：动态的，是程序的一次执行过程。

1. 同一个程序多次执行会对应多个进程。
2. ==进程是进程实体的运行过程，是系统进行资源分配和调度的一个独立单位==。
3. 进程由 PCB，程序段（代码：指令序列），数据段（运行过程中产生当各种数据）组成。
4. 其中 PCB 是给操作系统用的，程序段和数据段是给进程自己用的。

#### 进程的组成 -- PCB

==PCB==（Process Control Block）进程控制块，一个数据结构

1. 操作系统管理进程时所需要的信息，都会被放在 PCB 中。
2. PCB 是进程存在的唯一标志。
3. PID：当进程被创建时，操作系统会为该进程分配一个唯一的、不重复的“身份证号”。
4. ID（UID）：基本的进程描述信息，可以让操作系统区分各个进程。

|          操作系统          |        作用        |
| :--------------------: | :--------------: |
| 记录 PID 、UID（进程所属用户 ID） | 基本的进程描述信息，区分各个进程 |
|       记录分配给进程的资源       |      用于管理资源      |
|       记录了进程的运行情况       |   实现对进程的控制、调度    |

#### 进程的组成 -- 程序段、数据段

程序段、数据段、PCB 三部分组成了==进程实体==（==进程映像==） 

1. 引入进程实体的概念后，可把进程定义为： 
2. 进程是进程实体的==运行过程==，是系统进行==资源分配==和==调度==的一个独立单位
3. 一个进程被“调度”，就是指操作系统决定让这个进程上 CPU 运行
4. PCB 是进程存在的唯一标志
5. ==PCB==是给操作系统用的
6. 程序段、数据段是给进程自己用的
7. 进程是动态的，进程实体（进程映像）是静态的

### 进程的状态、转化、组织

#### 进程的状态

进程是程序的一次执行。在这个执行中，进程的状态会有各种变化。为了方便进程管理，把进程分为：运行态、就绪态、阻塞态、创建态、终止态五种状态。

1. 创建态：系统创建进程，操作系统给进程分配系统资源、PCB 等等
2. 就绪态：已经具备运行条件，等待空闲的 CPU ，进行调用
3. 运行态：当 CPU 处于空闲阶段就会在就绪态的进程里选择一个进行执行，使 CPU 进入运行态
4. 阻塞态：因为某个事件而暂时不可用运行
5. 终止态：运行进程从cpu撤销，操作系统就会回收资源、撤销PCB

一核的 CPU 一次只可以运行一个进程，多核的 CPU 一次可以运行多个进程。

### 作业和进程的区别

作业是用户需要计算机完成的某项任务。一个作业的完成要经过作业提交、作业收容、作业执行和作业完成四个阶段。而进程是对已提交完毕的程序所执行过程的描述。

1. 作业是用户向计算机提交任务的任务实体。
2. 在用户向计算机提交作业作业后，系统将它放入外存中的作业等待队列中等待执行。
3. 而进程则是完成用户任务的执行实体，是向系统申请分配资源的基本单位。
4. 任一进程，只要它被创建，总有相应的部分存在于内存中。
5. 一个作业可由多个进程组成，且必须至少由一个进程组成，反过来不成立。
6. 作业的概念主要用在批处理系统中，像 UNIX 这样的分时系统中就没有作业的概念。
7. 进程的概念则用在几乎所有的多道程序系统中。

### 进程控制

#### 进程控制的概念

1. ==进程控制就是要实现进程状态的转换==
2. 进程控制的主要功能是对系统中的所有进程实施有效的管理
3. 具有创建新进程、撤销已有进程、实现进程状态转换等功能

![](/images/os-notes/2.进程管理-20250504-150631.jpg)

####  进程控制的实现 -- 原语

1. 原语的执行具有原子性，运行在核心态，执行期间不允许被中断
2. 可以用 “==关中断指令==” 和 “==开中断指令==” 这两个特权指令实现原子性
3. CPU 执行 “关中断指令” 后，不再例行检查中断信号，直到执行 “开中断指令”
4. 正常情况：CPU 每执行完一条指令都会例行检查是否有中断信号需要处理

### 进程通信

#### 进程通信的概念

1. 进程通信就是指进程之间的信息交换 
2. 进程是分配系统资源的单位，因此各进程的内存地址空间相互独立
3. 为保证安全，一个进程不能直接访问另一个进程的地址空间

#### 进程通信 -- 共享存储

![](/images/os-notes/2.进程管理-20250513-235123.jpg)

1. 两个进程对共享空间的访问必须是==互斥==的
2. 互斥访问通过操作系统提供的工具实现
3. 操作系统只负责提供共享空间和同步互斥工具（如 P、V 操作）

- 基于数据结构的共享：
	1. 比如在共享空间里放一个长度为10的数组
	2. 这种共享方式速度慢、限制多，是一种低级通信方式

- 基于存储区的共享：
	1. 在内存中画出一块共享存储区
	2. ==数据的形式、存放位置都由进程控制==， 而不是操作系统
	3. 相比之下，这种共享方式速度更快，是一种==高级通信==方式

#### 进程通信 -- 管道通信

![](/images/os-notes/2.进程管理-20250513-235620.jpg)

1. 管道只能采用==半双工通信==，某一时间段内只能实现单向的传输
2. 如果要实现双向同时通信，则需要设置==两个管道== 
3. 各进程要==互斥地访问==管道
4. 数据以字符流的形式写入管道
5. 当==管道写满==，写进程的 write() 系统调用将被阻塞，等待读进程将数据取走
6. 当数据全部取走，==管道变空==，此时读进程的 read() 系统调用将被阻塞 
7. ==如果没写满，就不允许读；如果没读空，就不允许写== 
8. 数据一旦被读，就从管道中抛弃
9. 意味着==读进程最多只能有一个==，否则可能会有读错数据的情况

#### 进程通信 -- 消息传递

进程间的数据交换以==格式化的消息==（Message）为单位
进程通过操作系统提供的 “发送消息/接收消息” 两个==原语==进行数据交换

### 用户/内核态线程

####  用户级线程

![](/images/os-notes/2.进程管理-20250514-111007.jpg)

用户级线程是“代码逻辑”的载体 

1. 线程的管理工作都由用户空间中的==线程库==完成
2. 线程切换不需要 CPU 变态（不需要从用户态切换到内核态）
3. 操作系统不能意识到用户级线程的存在，操作系统只能感知到进程
4. 优点：线程切换无需内核介入，速度快
5. 缺点：一个线程阻塞（如系统调用）会导致整个进程阻塞；内核仅调度进程，无法将用户级线程映射到多个 CPU 核心（==真正的并行需要内核支持==）

####  内核级线程

![](/images/os-notes/2.进程管理-20250514-112139.jpg)

内核级线程是“运行机会”的载体，也是处理机分配的单位 

1. 内核级线程的管理工作（创建、调度、同步、销毁等）由==操作系统内核==完成
2. 线程切换需要 CPU 变态（ 用户态→内核态→用户态 ）
3. 操作系统可以意识到内核级线程，直接管理和调度内核级线程
4. 优点：可在多核 CPU 上同时运行多个线程（充分利用多核）
5. 缺点：每次线程切换都需要进入内核态，比用户级线程慢

- 操作系统会为内核级线程建立相应的 TCB（ThreadControlBlock，线程控制块）
- 通过 TCB 对线程进行管理

### 处理机调度概念、层次

#### 处理机调度

处理机调度：从就绪队列中按照算法将处理机分配给进程，以实现进程的并发执行

![](/images/os-notes/2.进程管理-20250506-173104.jpg)

####  高级调度（作业调度）

1. ==高级调度是辅存（外存）与内存之间的调度==
2. 分配内存等必要资源，并建立相应的进程（建立 PCB）
3. 每个作业调入一次，调出一次作业
4. 调入时会建立相应的 PCB，作业调出时撤销 PCB
5. 调入的时机需要操作系统确定，调出的时机为作业运行结束时  

####  中级调度（内存调度）
  
1. ==中级调度可将挂起状态的进程重新调入内存==
2. 引入了虚拟存储技术之后，可将暂时不能运行的进程调至外存等待
3. 合适时，再重新调入，可提高内存利用率和系统吞吐量
4. 暂时调到外存等待的进程状态为==挂起状态==
5. PCB 并不会一起调到外存，而是会常驻内存
6. PCB 中会记录进程数据在外存中的存放位置，进程状态等信息
7. 操作系统通过 PCB 来保持对各个进程的监控、管理
8. 被挂起的进程 PCB 会被放到挂起队列中

![](/images/os-notes/2.进程管理-20250514-160402.jpg)

####  低级调度（进程调度）
   
1. 按照某策略从就绪队列中选取一个进程，并分配处理机 
2. 进程调度是操作系统中最基本的一种调度
3. 进程调度的频率很高，一般几十毫秒一次

### 进程调度的时机、切换与过程、调度方式

####  进程调度的时机

![](/images/os-notes/2.进程管理-20250506-194801.jpg)

进程调度（低级调度），按照某种算法从就绪队列中选择一个进程为其分配处理机

1. 临界资源：一个时间段内只允许一个进程使用的资源，进程要互斥地访问临界资源
2. 临界区：访问临界资源的那段代码。

| 内核程序临界区             | 普通临界区            |
| ------------------- | ---------------- |
| 访问内核程序临界区时不能进行调度与切换 | 访问普通临界区时可进行调度与切换 |

#### 进程调度的方式

| 非剥夺调度方式                                 | 剥夺调度方式                                                 |
| :-------------------------------------- | :----------------------------------------------------- |
| 非抢占方式                                   | 抢占方式                                                   |
| 只允许进程主动放弃处理机<br>                        | 进程运行时，优先级更高的进程可介入                                      |
| 实现简单，系统开销小<br>无法及时处理紧急任务<br>适合于早期的批处理系统 | 可优先处理紧急进程，<br>可通过时钟中断实现各进程按时间片轮流执行<br>适合于分时操作系统、实时操作系统 |

####  进程的切换与过程

- “狭义的进程调度”与“进程切换”的区别： 
	1. 狭义的进程调度指的是从就绪队列中选中一个刚被暂停执行的进程		  
	2. 进程切换是指一个进程让出处理机，由另一个进程占用处理机的过程

广义的进程调度包含了选择一个进程和进程切换两个步骤

- 进程切换的过程主要完成了： 
	 1. 对原来运行进程各种数据的保存 
	 2. 对新的进程各种数据的恢复（如：程序计数器、程序状态字、各种数据寄存器等处理机现场信息，一般保存在进程控制块）

注意：频繁进行进程调度、切换，会使系统效率降低

### 调度算法的评价指标

####   调度算法的评价指标

![](/images/os-notes/2.进程管理-20250506-214915.jpg)

####  周转时间

周转时间：是指从作业被提交给系统，到作业完成为止的时间间隔

周转时间包括四个部分：
  1. 作业在外存后备队列上等待作业调度（高级调度）
  2. 进程在就绪队列上等待进程调度（低级调度）
  3. 进程在 CPU 上执行的时间（服务时间）
  4. 进程等待 I/O 操作完成的时间

####  带权周转时间

$$
\text{带权周转时间} = \frac{\text{周转时间}}{\text{实际运行时间（服务时间）}}
$$

实际运行时间：作业在 CPU 上实际执行的时间，不含等待时间

1. 带权周转时间越小越好，用户越满意  
2. 带权周转时间  >= 1
3. 若带权周转时间 = 1：表示作业没有等待，到达后立即执行（理想情况）
4. 若带权周转时间 > 1：数值越大，说明作业等待时间更长，调度效率更低

####  等待时间

等待时间：指进程/作业处于等待处理机状态时间之和

![](/images/os-notes/2.进程管理-20250507-001317.jpg)

1. 对于进程，等待时间就是指进程建立后==等待被服务的时间之和==
2. 等待 I/O 完成的期间其实进程也是在被服务的，所以==不计入等待时间==
3. ==对于作业==，要考虑建立进程后的等待时间，和作业在外存后备队列中的等待时间

####  响应时间

 响应时间：指从用户==提交请求==到==首次产生响应==所用的时间

####  短作业优先（SJF,Shortest JobFirst）

### 进程同步与互斥

#### 进程同步

读进程和写进程并发地运行，并发必然导致异步性。因此“写数据”和“读数据”两个操作执行的先后顺序是不确定的。而实际应用中，又必须按照“写数据→读数据”的顺序来执行的。

- 如何解决这种异步问题，就是“进程同步”所讨论的内容。

![](/images/os-notes/2.进程管理-20250527-195853.jpg)

1. ==同步亦称直接制约关系==
2. 同步指为完成某种任务而建立的多个进程为协调它们之间的工作次序而产生的制约关系
3. 进程间的直接制约关系就是源于它们之间的相互合作

#### 进程互斥

1. 我们把一个时间段内只允许一个进程使用的资源称为临界资源
2. 许多物理设备、变量、数据、内存缓冲区等都属于临界资源
3. 对临界资源的访问，必须互斥地进行
4. ==互斥，亦称间接制约关系==
5. 进程互斥指当一个进程访问某临界资源时，另一个想访问该临界资源的进程必须等待
6. 当前访问临界资源的进程访问结束且释放该资源后，另一个进程才能去访问临界资源

![](/images/os-notes/2.进程管理-20250527-200810.jpg)

#### 互斥原则

对临界资源的互斥访问，可以在逻辑上分为如下四个部分：

![](/images/os-notes/2.进程管理-20250527-201014.jpg)

为了实现对临界资源的互斥访问，同时保证系统整体性能，需要遵循以下原则：

1. 空闲让进：临界区空闲时，可以允许一个请求进入临界区的进程立即进入临界区
2. 忙则等待：当已有进程进入临界区时，其他试图进入临界区的进程必须等待
3. 有限等待：对请求访问的进程，应保证能在有限时间内进入临界区（保证不会饥饿）
4. 让权等待：当进程不能进入临界区时，应立即释放处理机，防止进程忙等待

### 进程互斥的软件实现方法

#### 单标志法

单标志法通过一个共享的标志变量来控制进程间的交替执行，确保其不会同时进入临界区

- 算法思想：
	1. 设置一个共享的标志变量（通常是一个布尔值或整型）
	2. 当一个进程完成其临界区操作后，会修改标志以允许另一个进程进入
	3. 另一个进程在进入临界区前会检查这个标志

```C
int turn = 0;  //turn 表示当前允许进入临界区的进程号

P0 进程:
	while (turn != 0);   //1.进入区
	critical section;    //2.临界区
	turn = 1;            //3.退出区
	remainder section;   //4.剩余区

P1 进程:
	while (turn != 1);   //5.进入区
	critical section;    //6.临界区
	turn = 0;            //7.退出区
	remainder section;   //8.剩余区
```

- “int turn = 0”，表明当前允许 P<sub>0</sub> 进入临界区，但如果 P<sub>1</sub> 先获得 CPU ，则会：
	1. P<sub>1</sub> 会在代码 `while (turn != 1)` 处自旋等待；因为 `turn = 0`，不满足进入条件
	2. 当 P<sub>1</sub> 的时间片用完，CPU 调度切换到 P<sub>0</sub>
	3. P<sub>0</sub> 检查 `turn == 0`，可以顺利进入临界区
	4. 当 P<sub>0</sub> 退出临界区并执行代码 `turn = 1` 后，P<sub>1</sub> 才能继续执行

1. 当临界区是空闲的，P<sub>1</sub> 仍然必须等待 P<sub>0</sub> 主动让出权限才能正常运行。
2. 违反了 “==空闲让进==（No busy waiting）”原则，即当临界区未被占用时，任何请求进入的进程都应能立即访问，而不必依赖其他进程的配合。

#### 双标志先检查法（Two-Flag Check-First Method）

双标志先检查法通过两个标志变量来协调两个进程的互斥访问，避免竞争条件

- 算法思想：
	1. 每个进程维护一个自己的标志“flag \[ i ]”，用来表示自己是否想进入临界区 
	2. 在进入临界区之前，进程先检查对方进程的标志：
		1. 如果对方不想进入临界区“flag \[ j ] == false”，则自己可以进入
		2. 如果对方也想进入“flag \[ j ] == true”，则等待
	3. 通过设置一个布尔型数组 flag\[]，采用“先检查后上锁”的方法

```C
bool flag [2];     //表示进入临界区意愿的数组
flag [0] = false;
flag [1] = false;  //刚开始设置为两个进程都不想进入临界区

P0 进程:
	while (flag[1]);    // 检查 P1 是否想进入临界区（忙等待）
	flag [0] = true;    // 标记 P0 想进入临界区
	critical section;   // 访问临界区
	flag [0] = false;   // 退出临界区，释放访问权
	remainder section;

P1 进程:
	while (flag[0]);    // 检查 P0 是否想进入临界区（忙等待）
	flag [1] = true;    // 标记 P1 想进入临界区
	critical section;   // 访问临界区
	flag [1] = false;   // 退出临界区，释放访问权
	remainder section;
```

如果按照“P<sub>0</sub> 检查 → P<sub>1</sub>检查 → P<sub>0</sub>设置 → P<sub>1</sub>设置 → P<sub>0</sub>进入临界区 → P<sub>1</sub>进入临界区”的顺序，则会使得 P<sub>0</sub>、P<sub>1</sub> 同时进入临界区，违反了“==忙着等待==”的原则。

- 原因
	1. 进入区的“检查”和“上锁”不是原子操作，中间可被中断
	2. 缺乏互斥机制，无法保证检查标志和设置标志之间不被其他进程干扰

#### 双标志后检查法

双标志后检查法类似于双标志先检查法，但调整了标志设置和检查的顺序

- 算法思想：
	1. 每个进程维护一个自己的标志`flag[i]`，表示自己是否想要进入临界区
	2. 与双标志先检查法的区别：
		1. 先检查对方进程的标志，再设置自己的标志
		2. 如果对方已经设置了标志`flag[j] == true`，则等待
		3. 否则，设置自己的标志并进入临界区
	3. 采用“先上锁后检查”的方法，避免两个进程同时进入临界区的问题

```C
bool flag[2] = {false, false};  // 初始时两个进程都不想进入临界区

flag[0] = true;     // 先声明 P0 想进入临界区
while (flag[1]);    // 如果 P1 也想进入，则循环等待
critical section;   // 访问临界区
flag[0] = false;    // 退出临界区，释放访问权
remainder section;

flag[1] = true;     // 先声明 P1 想进入临界区
while (flag[0]);    // 如果 P0 也想进入，则循环等待
critical section;   // 访问临界区
flag[1] = false;    // 退出临界区，释放访问权
remainder section;
```

如果按照“P<sub>0</sub> 设置 → P<sub>1</sub>设置 → P<sub>0</sub>检查 → P<sub>1</sub>检查“的顺序，会导致 P<sub>0</sub> 和 P<sub>1</sub> 互相等待对方释放标志，导致死锁（Deadlock），两者都无法进入临界区，违反了“==空闲让进==”和“==有限等待==”。

- 原因 
	1. “先声明，后检查”的策略导致了“相互谦让”变成“互相阻塞”
	2. 缺乏仲裁机制（如 turn 变量）来决定谁应该优先进入

#### Peterson算法

- 算法思想：
	1. 通过双标志 + 轮转变量来解决同步问题
	2. 用 Flag\[2] 表示进程是否想进入临界区（类似双标志法）
	3. 用 Turn 表示当前应该轮到谁进入（解决“互相谦让”导致的死锁）

```C
bool flag[2] = {false, false}; // 进程进入意愿
int turn = 0;                  // 轮转变量

P0 进程
void P0() {
    flag[0] = true;                // 1. P0声明想进入
    turn = 1;                      // 2. 主动让给P1
    while (flag[1] && turn == 1);  // 3. 检查P1是否想进且是否轮到P1
    critical_section();            // 4. 进入临界区
    flag[0] = false;               // 5. 退出临界区
}

P1 进程
void P1() {
    flag[1] = true;                // 1. P1声明想进入
    turn = 0;                      // 2. 主动让给P0
    while (flag[0] && turn == 0);  // 3. 检查P0是否想进且是否轮到P0
    critical_section();            // 4. 进入临界区
    flag[1] = false;               // 5. 退出临界区
}
```

1. 通过引入 turn 变量强制轮流进入，解决“双标志后检查法”的死锁问题
2. 同时满足了“空闲让进”、“忙则等待”、“有限等待”三个原则
3. 但是依然未遵循“让权等待”的原则

### 进程互斥的硬件实现方法

#### 中断屏蔽方法

进程切换利用 “开/关中断指令” 实现

1. 关中断后不允许当前进程被中断，也不会发生进程切换
2. 直到当前进程访问完临界区，再执行开中断指令，别的进程才可访问临界区

缺点：不适用于多处理机，不适用于用户进程（因为开/关中断指令只运行在内核态）

####  TestAndSet 指令（TS / TSL 指令）

TestAndSet 是一个不可中断的原子指令（由 CPU 直接支持）

- 功能：
	1. 读取某个内存位置的值
	2. 设置该内存位置为一个新值（通常为 turn ）
	3. 返回旧值

```cpp
bool TestAndSet(bool *lock) {      // TestAndSet 的原子操作（硬件实现）
    bool old = *lock;  // 读取旧值
    *lock = true;      // 设置新值（上锁）
    return old;        // 返回旧值
}

bool lock = false;  // 初始时锁是释放状态

while (TestAndSet(&lock)) {  // 尝试获取锁
    // 忙等待（Busy Waiting），直到锁被释放
}
// 临界区（Critical Section）
lock = false;  // 释放锁
// 剩余区（Remainder Section）
```

- 工作原理  
	1. 初始状态：“lock = false”（锁是释放的）
	2. 进程尝试获取锁：
		1. 调用"TestAndSet (&lock) "，返回 lock 的值 false，设置 lock=true
		2. 由于返回 false, while 循环不执行，进程进入临界区
	3. 此时如果有别的进程尝试获取锁：
		1. "TestAndSet (&lock)"会返回 true（因为锁已被占用）
		2. 进程会在 while 循环中''忙等''，直到锁被释放
	4. 释放锁：进程退出临界区时，会设置 lock=false，允许别的进程进入

####  Swap 指令（Exchange 指令、XCHG 指令）

Swap 指令与 TestAndSet 类似，但采用交换内存值的方式实现互斥
Swap 指令是原子操作（由 CPU 直接支持），是一种基础的同步原语

- 功能：
	1. 返回某个内存位置的值和寄存器的值
	2. 返回内存位置的旧值

1. Swap 指令是用硬件实现的，执行的过程不允许被中断
2. 如果锁被占用，线程会持续循环检查（自旋），直到锁释放，即''忙等''
3. 实现简单，适用于多处理机环境

```CPP
bool Swap(bool *a, bool *b) {   // Swap 的原子操作（硬件实现）
    bool tmp = *a;
    *a = *b;
    *b = tmp;
    return tmp;        // 返回 a 的旧值
}

bool lock = false;     // 初始时锁是释放的

bool key = true;       // 本地变量
while (key == true) {
    Swap(&lock, &key); // 原子交换 lock 和 key
}
// 临界区（Critical Section）
lock = false;       // 释放锁
// 剩余区（Remainder Section）
```

- 工作原理：
	1. 初始状态：lock = false（锁是释放的），key = true（本地变量）
	2. 进程尝试获取锁：
		1. 调用 Swap(&lock, &key)，该指令原子地执行以下操作：
			 1. 交换 lock 和 key 的值
			 2. Lock 变成 true（表示锁被占用）
			 3. Key 变成 lock 的旧值（false）
		2. 由于 key 变为 false，while 循环条件不成立，进程退出循环并进入临界区
	4. 如果另一个进程尝试获取锁：
		1. 此时 lock = true（锁已被占用）
		2. 调用 Swap(&lock, &key) 后：
			  1. Lock 保持 true（因为锁已被占用）。
			  2. key 被设置为 true（因为 lock 的旧值是 true）  
		3. 由于 key 为 true，while 循环继续执行，进程进入忙等待，直到锁被释放
	5. 释放锁：
		1. 当前进程退出临界区时，将 lock 设置为 false（释放锁）
		2. 其他正在忙等待的进程在下一次 Swap 时会发现 lock = false，从而成功获取锁并进入临界区
 
### 信号量机制

#### 信号量机制的概念

信号量的值 = 这种资源的剩余数量
（信号量的值如果小于 0，说明此时有进程在等待这种资源）

1. ==信号量是一个变量，可用信号量来表示系统中资源的数量==
2. 用户进程可通过一对原语来对信号量进行操作，从而实现进程互斥、进程同步
3. 原语是由 ''关中断/开中断指令'' 实现的
4. 一对原语：wait (参数) 、signal (参数)
5. wait(S)、signal(S) 两个操作可简写为 P(S)、V(S) ，表''申请''、''释放"

P(S) —— 申请一个资源 S，如果资源不够就阻塞等待 
V(S) —— 释放一个资源 S，如果有进程在等待该资源，则唤醒一个进程

####  信号量机制 -- 整型信号量

用一个整数型的变量作为信号量，用来表示系统中某种资源的数量

与普通整数变量的区别：对信号量的操作只有三种， 即初始化、P 操作、V 操作

```cpp
void wait (int S) {    //wait 原语，相当于“进入区”
	while (S <= 0);    //如果资源数不够，就一直循环等待
	S=S-1;             //如果资源数够，则占用一个资源
}

void signal (int S) {  //signal 原语，相当于“退出区”
	S=S+1;             //使用完资源后，在退出区释放资源
}
```

缺点：存在 “忙等” 问题

####  信号量机制 -- 记录型信号量

用记录型数据结构表示的信号量

```cpp
//记录型信号量的定义
typedef struct {
	int value;          //剩余资源数
	struct process *L;  //等待队列
} semaphore;

//某进程需要使用资源时候，通过 wait 原语申请
void wait (semaphore S ) {
	S.value--;
	if (S.value < 0) {
		block (S.L);   
	}
}

//进程使用完资源后，通过 signal 原语释放
void signal (semaphore S) {
	s.value++;
	if (S.value <= 0) {
		wakeup (S.L);
	}
}
```

wait 原语

1. 如果剩余资源数不够，使用 block 原语==使进程从运行态进入阻塞态==
2. 并挂到信号量S的等待队列（即阻塞队列）中
3. 遵循了“让权等待”原则，不会出现“忙等”现象

signal 原语

1. 释放资源后，若还有别的进程在等待资源，
2. 则使用 wakeup 原语唤醒等待队列中的一个进程，==该进程从阻塞态变为就绪态==

### 用信号量机制实现进程互斥、同步、前驱关系

#### 实现进程互斥

- 用信号量实现进程互斥：
	1. 分析并发进程的关键活动，划定临界区
	2. 设置互斥信号量 mutex ，初值为1 
	3. 在进入区 P(mutex) —— 申请资源
	4. 在退出区 V(mutex) —— 释放资源

注意：对不同的临界资源需要设置不同的互斥信号量 

- P、V 操作必须成对出现
	1. 缺少 P(mutex) 就不能保证临界资源的互斥访问
	2. 缺少 V(mutex) 会导致资源永不被释放，等待进程永不被唤醒

#### 实现进程同步

进程同步：要让各并发进程按要求有序地推进

- 用信号量实现进程同步： 
	1. 分析什么地方需要实现 “同步关系”，保证 “==一前一后==” 执行的两个操作
	2. 设置同步信号量 S ，初始为0 
	3. 在 “前操作” 之后执行 V(S) 
	4. 在 “后操作” 之前执行 P(S)

#### 实现前驱关系

每一对前驱关系都是一个进程同步问题

- 需要保证一前一后的操作，因此：
	1. 要为每一对前驱关系各设置一个同步信号量 
	2. 在“前操作”之后对相应的同步信号量执行 V 操作 
	3. 在“后操作”之前对相应的同步信号量执行 P 操作

### 生产者 - 消费者问题

#### 生产者 - 消费者的概念

一组生产者向共享缓冲区生产数据，另一组消费者从中消费数据

1. 系统中有一组生产者进程和一组消费者进程 
2. 生产者、消费者共享一个初始为空、大小为 N 的缓冲区
3. 缓冲区没满 == 生产者生产
4. 缓冲区没空 == 消费者消费
5. 缓冲区是临界资源，各进程必须==互斥==地访问
6. 实现互斥是在同一进程中进行一对 P V 操作

![](/images/os-notes/2.进程管理-20250508-113853.jpg)

```c
semaphore mutex = 1;  //互斥信号量，实现对缓冲区的互斥访问 
semaphore empty = n;  //同步信号量，表示空闲缓冲区的数量 
semaphore full  = 0;  //同步信号量，表示产品的数量，也即非空缓冲区的数量


producer () { 
	while(1){ 
		//生产一个产品
		P (empty);       //消耗一个空闲缓冲区
		P (mutex); 
		//把产品放入缓冲区
		V (mutex); 
		V (full);        //增加一个产品
	} 
}


consumer(){ 
	while(1){ 
		P (full);    //消耗一个产品（非空缓冲区）
		P (mutex); 
		//从缓冲区取出一个产品
		V (mutex); 
		V (empty);   //增加一个空闲缓冲区
		使用产品;
	}
}
```

#### 能否改变相邻 P、V 操作的顺序？

![](/images/os-notes/2.进程管理-20250508-115346.jpg)

缓冲区放满产品，则 empty=0，full=N 

1. 生产者进程执行 ① 使 mutex 变0，再执行 ②
2. 因没有空闲缓冲区，生产者被阻塞 
3. 由于生产者阻塞，因此切换回消费者进程
4. 消费者进程执行 ③   
5. 由于 mutex 为0，生产者未释放对临界资源的“锁”，消费者也被阻塞
6. 造成生产者等待消费者释放空闲缓冲区，消费者又等待生产者释放临界区
7. 生产者和消费者循环等待被对方唤醒，出现“死锁”

缓冲区中没有产品，即 full=0，empty=N

1. 按 ③ ④ ① 的顺序执行也发生死锁
2. 因此，实现互斥的 P 操作一定要在实现同步的 P 操作之后 
3.  V 操作不会导致进程阻塞，可以交换两个 V 的操作顺序

### 多生产者 - 多消费者

多个生产者线程向共享缓冲区添加数据，同时多个消费者线程从中取出数据

1. 如果缓冲区大小 = 1，有时可不设置互斥信号量就可以实现互斥访问缓冲区的功能
2. 如果缓冲区大小 > 1，必须专门设置一个互斥信号量 mutex 来保证互斥访问缓冲区
3. 默认可以加上互斥信号量，保证各进程一定会互斥地访问缓冲区
4. 实现互斥的 P 操作一定要在实现同步的 P 操作之后，否则可能引起“死锁”

### 哲学家进餐问题

#### 哲学家进餐问题的概念

### 管程

#### 引入管程的原因

信号量机制存在的问题：编写程序困难、易出错
能不能设计一种机制，让程序员不需要再关注复杂的 PV 操作，让写代码更轻松呢？

1973年，BrinchHansen在程序设计语言 (Pascal) 中引入了“管程”，一种高级同步机制

####  管程的定义

管程是一种特殊的软件模块，由这些部分组成： 

1. 局部于管程的==共享数据结构==说明
2. 对该数据结构进行操作的==一组过程（函数）==
3. 对局部于管程的共享数据设置初始值的语句
4. 管程有一个名字。 跨考Tips:“过程”其实就是“函数‘’

####  管程的基本特征

1. 局部于管程的数据只能被局部于管程的过程所访问
2. 一个进程只有通过调用管程内的过程才能进入管程访问共享数据
3. ==每次仅允许一个进程在管程内执行某个内部过程==

#### 拓展 1：用管程解决生产者消费者问题

![](/images/os-notes/2.进程管理-20250509-144835.jpg)

引入管程的目的就是要更方便地实现进程互斥和同步

1. 在管程中定义共享数据（如生产者-消费者问题的缓冲区） 
2. 在管程中定义用于访问这些共享数据的“入口”，即函数
	（如生产者消费者问题中，可以定义放入函数、取出函数来管理缓存区） 
3. 只有==通过这些特定的“入口”才能访问共享数据== 
4. 管程中有很多“入口”，但每次==只开放其中一个“入口”，并只让一个进程或线程进入==
	1. 如生产者消费者问题中，各进程需要互斥地访问共享缓冲区
	2. 管程的这种特性即可保证一个时间段内最多只会有一个进程在访问缓冲区
	3. ==这种互斥特性是由编译器负责实现的， 程序员不用费心==
5. 可在管程中设置==条件变量及等待/唤醒操作==以解决同步问题，也可让一个进程或线程在条件变量上等待（此时，该进程应先释放管程的使用权，也就是让出“入口”）；可以通过唤醒操作将等待在条件变量上的进程或线程唤醒

- 程序员可提前定义一个管程，方便他人利用这个管程来实现进程同步/互斥
- 体现 “封装” 思想

### 死锁

#### 死锁、饥饿、死循环

1. 死锁：各进程互相等待对方手里的资源，导致各进程都阻塞，无法向前推进的现象。
2. 饥饿：由于长期得不到想要的资源，某进程无法向前推进的现象。
3. 死循环：某进程执行过程中一直跳不出某个循环的现象。

#### 死锁产生的必要条件

产生死锁必须同时满足一下四个条件，只要其中任一条件不成立，死锁就不会发生。

1. 互斥条件：只有对必须互斥使用的资源的争抢才会导致死锁。
2. 不剥夺条件：进程所获得的资源在未使用完之前，不能被其他进程强行夺走，只能主动释放。
3. 请求和保持条件：进程已经保持了至少一个资源，但又提出了新的资源请求，而该资源又被其他进程占有，此时请求进程被阻塞，但又对自己已有的资源保持不放。
4. 循环等待条件：存在一种进程资源的循环等待链，链中的每一个进程已获得的资源同时被下一个进程所请求。

- 注意：发生死锁时一定有循环等待，但是发生循环等待时未必死锁。
- 循环等待是死锁的必要不充分条件。
- 如果同类资源数大于 1，则即使有循环等待，也未必发生死锁。
- 如果系统中每类资源都只有一个，那循环等待就是死锁的充分必要条件了。

#### 什么时候会发生死锁

- 对系统资源的竞争
	1. 各进程对不可剥夺的资源（如打印机）的竞争可能引起死锁
	2. 对可剥夺的资源（CPU）的竞争是不会引起死锁的
- 进程推进顺序非法
	1. 请求和释放资源的顺序不当，也同样会导致死锁
	2. 例如，并发执行的进程 P<sub>1</sub>、P<sub>2</sub>  分别申请并占有了资源 R<sub>1</sub>、R<sub>2</sub>
	3. 之后进程 P<sub>1</sub> 又紧接着申请资源 R<sub>2</sub> ，而进程 P<sub>2</sub> 又申请资源 R<sub>1</sub> 
	4. 两者会因为申请的资源被对方占有而阻塞，从而发生死锁
- 信号量的不当使用
	1. 在生产者-消费者问题中
	2. 如果实现互斥的 P 操作在实现同步的 P 操作之前，就有可能导致死锁
	3. 可以把互斥信号量、同步信号量也看做是一种抽象的系统资源

#### 死锁的处理策略

1. 预防死锁：破坏死锁产生的四个必要条件中的一个或几个
2. 避免死锁：用某种方法防止系统进入不安全状态，从而避免死锁（银行家算法)
3. 死锁的检测和解除：通过操作系统检测出死锁的发生，再采取某种措施解除死锁

### 死锁的处理策略--预防死锁

![](/images/os-notes/2.进程管理-20250509-150851.jpg)

####  破坏互斥条件

互斥条件：只有对必须互斥使用的资源的争抢才会导致死锁

- 可使用 ==SPOOLing== 技术，立即接收各进程对资源的使用请求，不用阻塞等候
- 缺点：不是所有资源都可改造成可共享使用的资源，大多数情况还需要保护互斥性

####  破坏不剥夺条件

不剥夺条件：进程所获得的资源若未使用完，不能被其他进程夺走，只能主动释放

- 破坏不剥夺条件的==方法==： 
	1. 当某个进程请求新的资源得不到满足时，它必须立即释放保持的所有资源
	2. 当某个进程所需资源被其他进程所占有，可由操作系统协助将所需资源强行剥夺     
- 缺点： 
	1. 实现起来比较复杂
	2. 可能造成前一阶段工作失效，只适用于易保存和恢复状态的资源，如CPU
	3. 反复地申请和释放资源会增加系统开销，降低系统吞吐量
	4. 方法一容易导致进程饥饿，使得进程在获取资源和释放资源间往返

####  破坏请求和保持条件

请求和保持条件：进程已保持至少一个资源，又提出了新的资源请求，而该资源又被其他进程占有，此时请求进程被阻塞，但对自己已有的资源保持不放

- 可以采用==静态分配方法==，即进程在运行前一次申请完它所需要的全部资源
- 缺点： 有些资源可能只需要使用较短时间，如果进程运行期间都一直保持着所有资源，会造成严重的资源浪费，==资源利用率极低==；有可能==导致某些进程饥饿==

####  破坏循环等待条件

循环等待条件：存在一种进程==资源的循环等待链==，链中的每一个进程已获得的资源同时被下一个进程所请求。 

- 可采用==顺序资源分配法==，给系统中的资源编号，规定每个进程==必须按编号递增的顺序请求资源==， 同类资源（即编号相同的资源）一次申请完 
- 原理分析：一个进程只有已占有小编号的资源时，才有资格申请更大编号的资源。按此规则，已持有大编号资源的进程不可能逆向地回来申请小编号的资源，从而就不会产生循环等待的现象。
- 缺点： 
  1. 不方便增加新的设备，因为可能需要重新分配所有的编号
  2. 进程实际使用资源的顺序可能和编号递增顺序不一致，会导致资源浪费 
  3. 必须按规定次序申请资源，用户编程麻烦

### 死锁的处理策略--避免死锁

![](/images/os-notes/2.进程管理-20250509-155921.jpg)

####  安全序列

概念：如果系统按照某种序列分配资源，每个进程都能顺利完成

1. ==只要能找出一个安全序列==，系统就是安全状态
2. 安全序列可能有多个
3. 如果分配资源后，系统中找不出任何一个安全序列，系统就进入了不安全状态
4. 不安全状态意味着所有进程可能都无法顺利的执行下去
5. 除非有进程提前归还资源，使得系统进入安全状态
6. 如果系统处于安全状态，就一定不会发生死锁
7. 系统处于不安全状态未必死锁，但==死锁时一定处于不安全状态==

####  “银行家算法”的核心思想

资源分配前先判断分配是否会导致系统进入不安全状态，再决定是否答应资源分配请求

1. ‘’银行家算法‘’可以避免死锁
2. 通过循环检查将进程加入到安全序列，称为安全性算法
3. 每一轮检查从编号较小的进程开始检查，可以更快速地得到安全序列

####  “银行家算法”的数据结构

1. 长度为 M 的一维数组 Available 表示还有多少可用资源 
2. N\*M 矩阵 Max 表示各进程对资源的最大需求数 
3. N\*M 矩阵 Allocation 表示已经给各进程分配了多少资源 
4. Max – Allocation = Need 矩阵表示各进程最多还需要多少资源 
5. 用长度为 M 的一位数组 Request 表示进程此次申请的各种资源数 
 
####  银行家算法步骤 
  
1. 检查此次申请是否超过了之前声明的最大需求数 
2. 检查此时系统剩余的可用资源是否还能满足这次请求 
3. 试探着分配，更改各数据结构 
4. 用安全性算法检查此次分配是否会导致系统进入不安全状态   
    
####  安全性算法步骤
 
1. 检查当前的剩余可用资源是否能满足某个进程的最大需求
2. 如果可以，就把该进程加入安全序列， 并把该进程持有的资源全部回收 
3. 不断重复上述过程，看最终是否能让所有进程都加入安全序列

### 死锁的处理策略--检测和解除

![](/images/os-notes/2.进程管理-20250510-094237.jpg)

####  死锁的检测

死锁检测算法：用于检测系统状态，以确定系统中是否发生了死锁

1. 用==某种数据结构==来保存资源的请求和分配信息
2. 提供==一种算法==，利用上述信息来检测系统是否已进入死锁状态

![](/images/os-notes/2.进程管理-20250510-094749.jpg)

![](/images/os-notes/2.进程管理-20250510-094736.jpg)

1. 如果系统中剩余的可用资源数足够满足进程的需求，那么进程暂时不会阻塞
2. 当进程执行结束，归还系统资源，可能使某些正在等待资源的进程被激活并执行
3. 这些被激活的进程执行完后又会归还资源，可能又会激活另外一些阻塞的进程
4. 这样可以消除所有的边，说明未发生死锁
5. 如果按上述过程分析，最终能==消除所有边==，就称这个==图是可完全简化==的
6. 此时一定==没有发生死锁==（相当于能找到一个安全序列）
7. 如果最终==不能消除所有边==，那么此时就是==发生了死锁==
8. 最终还连着边的进程就是处于死锁状态的进程

####  死锁的解除

死锁解除算法：已发生死锁时，利用该算法可将系统从死锁状态中解脱出来

- 解除死锁的主要方法有：
	1. 资源剥夺法：挂起某些死锁进程（暂时放到外存上），并抢占它的资源，将资源分配给其他的死锁进程。同时防止被挂起的进程长时间得不到资源而饥饿 
	2. 撤销进程法（终止进程法）：强制撤销部分甚至全部死锁进程，并剥夺这些进程的资源。实现简单，可能使得快运行完的进程被迫中止
	3. 进程回退法：让一个或多个死锁进程回退到足以避免死锁的地步。这就要求系统要记录进程的历史信息，设置还原点

一旦检测出死锁的发生，就应该立即解除死锁

![](/images/os-notes/2.进程管理-20250510-103348.jpg)

- 如何决定解除 “哪个进程”
	1. 进程优先级
	2. 已执行多长时间
	3. 还要多久能完成
	4. 进程已经使用了多少资源
