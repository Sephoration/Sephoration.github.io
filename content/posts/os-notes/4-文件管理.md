### 初识文件管理    

#### 知识总览

文件——就是一组有意义的信息/数据集合

![](/images/os-notes/4.文件管理-20250518-000017.jpg)

1. 一个文件有哪些属性？ 
2. ==文件内部==的数据应该怎样组织起来？
3. ==文件之间==应该又应该怎么组织起来？
4. 从下往上看，OS应提供哪些功能，才能方便用户、应用程序使用文件？
5. 从上往下看，文件数据应该怎么存放在外存（磁盘）上？

#### 文件的属性

一个文件有哪些属性？ 

- 文件名：同一目录下不允许有重名文件 
- 标识符：一个系统内的各文件标识符唯一，操作系统用于区分各个文件的内部名称
- 类型：指明文件的类型 
- 位置：文件存放的路径（让用户使用）、在外存中的地址（让操作系统使用） 
- 大小：指明文件大小 
- 创建时间、上次修改时间 
- 文件所有者信息 
- 保护信息：对文件进行保护的访问 
- 控制信息

#### 文件内部的数据怎样组织起来？

1. 无结构文件（如文本文件）—— 由一些二进制或字符流组成，又称“流式文件”
2. 有结构文件（如数据库表）—— 由一组相似的记录组成，又称“记录式文件”
3. 数据项是文件系统中==最基本的数据单位==
4. 记录是一组相关==数据项的集合==

![](/images/os-notes/4.文件管理-20250518-001253.jpg)

- 有结构文件中，各个记录间应该如何组织
	1. 应该顺序存放？ 
	2. 还是用索引表来表示记录间的顺序？
	3. 这是“文件的逻辑结构” 重点要探讨的问题

#### 文件之间应该怎样组织起来？

![](/images/os-notes/4.文件管理-20250518-115107.jpg)

1. 用户可以自己创建一层一层的目录，各层目录中存放相应的文件
2. 系统中的各个文件就通过一层一层的目录合理有序的组织起来了
3. 目录其实也是一种特殊的有结构文件（由记录组成）
4. 所谓的 “目录” 其实就是我们熟悉的“文件夹”
5. 如何实现文件目录是之后会重点探讨的问题

#### 操作系统应该向上提供哪些功能？

|  功能  |  系统调用  |                     |
| :--: | :----: | :-----------------: |
| 创建文件 | Create |    由图形化交互进程在背后调用    |
| 读文件  |  Read  | 将文件数据由外存读入内存，让CPU处理 |
| 写文件  | Write  |  将更改过的文件数据从内存写回外存   |
| 删除文件 | Delete |     将文件数据从外存中删除     |
| 打开文件 |  Open  |   读/写文件之前，需要“打开文件   |
| 关闭文件 | Close  |  读/写文件结束之后，需要“关闭文件  |

#### 从上往下看，文件应如何存放在外存？

- 操作系统
	1. 以 “块” 为单位为文件分配存储空间，
	2. 需要将文件的逻辑地址（逻辑块号，块内地址）
	3. 转换为外存的物理地址（物理块号，块内地址）

外存分成一个个存储单元，每个存储单元对应一个物理地址

1. 类似于“内存块”，外存分为一个个“块/磁盘块/物理块”
2. 每个磁盘块的大小是相等的
3. 块内地址的位数取决于磁盘块的大小
4. 外存中的数据读入内存时同样以“块”为单位
5. 因此即使一个文件大小只有 10B，但它依然需要占用 1KB 的磁盘块

#### 其他需要操作系统实现的文件管理功能

文件共享：使多个用户可以共享使用一个文件
文件保护：如何保证不同的用户对文件有不同的操作权限

### 文件的逻辑结构

#### 知识总览

![](/images/os-notes/4.文件管理-20250518-141032.jpg)

“逻辑结构”，在用户看来， 文件内部的数据应该是如何组织起来的。
“物理结构”，在操作系统看来， 文件的数据是如何存放在外存中的。

算法的具体实现与逻辑结构、物理结构都有关
文件操作的具体实现与文件的逻辑结构、物理结构都有关

#### 无结构文件

无结构文件：文件内部的数据就是一系列二进制流或字符流组成，又称“流式文件”

1. 文件内部的数据是一系列字符流，没有明显的结构特性
2. 因此也不用探讨无结构文件的“逻辑结构”问题
3. 按文件是否有结构分类，可以分为无结构文件、有结构文件两种

#### 有结构文件

有结构文件：由一组相似的记录组成，又称“记录式文件”

1. 每条记录由若干个数据项组成
2. 如： 数据库表文件
3. 一般来说，每条记录有一个数据项可作为关键字
4. 根据各条记录的长度（占用的存储空间），可分为==定长记录==和==可变长记录==

#### 有结构文件的逻辑结构

![](/images/os-notes/4.文件管理-20250518-125013.jpg)

#### 顺序文件

- 顺序文件：
	1. 文件中的记录一个接一个地顺序排列（逻辑上），
	2. 记录可以是定长的或可变长的，
	3. 各个记录在物理上可以顺序存储或链式存储。

1. 定长记录的顺序文件，若物理上采用顺序存储，则可实现随机存取
2. 若能再保证记录的顺序结构，则可实现快速检索（根据关键字快速找到记录）
3. 顺序文件的缺点是增加/删除一个记录比较困难（如果是串结构则相对简单）
4. 一般来说，考试题目中所说的“顺序文件”指的是物理上顺序存储的顺序文件

![](/images/os-notes/4.文件管理-20250518-125141.jpg)

![](/images/os-notes/4.文件管理-20250518-125205.jpg)

![](/images/os-notes/4.文件管理-20250518-125604.jpg)

![](/images/os-notes/4.文件管理-20250518-125400.jpg)

#### 索引文件

对于可变长记录文件，要找到第 i 个记录，必须先顺序第查找前 i-1 个记录， 
但是很多应用场景中又必须使用可变长记录。如何解决这个问题？

1. 建立一张索引表以加快文件检索速度。
2. 每条记录对应一个索引项。
3. 文件中的这些记录在物理上可以离散地存放。

![](/images/os-notes/4.文件管理-20250518-130504.jpg)

1. 索引表本身是定长记录的顺序文件
2. 因此可以快速找到第 i 个记录对应的索引项
3. 可将关键字作为索引号内容
4. 若按关键字顺序排列，则还可以支持按照关键字折半查找
5. 每当要增加/删除一个记录时，需要对索引表进行修改
6. 由于索引文件有很快的检索速度，主要用于对信息处理的及时性要求较高的场合

- 另外，可以用不同的数据项建立多个索引表
	1. 如：学生信息表中，可用关键字“学号”建立一张索引表
	2. 也可用“姓名”建立一张索引表，这样就可以根据“姓名”快速地检索文件
	3. Eg：SQL就支持根据某个数据项建立索引的功能

#### 索引顺序文件

- 思考索引文件的缺点：每个记录对应一个索引表项，因此索引表可能会很大 
	1. 比如：文件的每个记录平均只占 8B，而每个索引表项占32个字节
	2. 那么索引表都要比文件内容本身大4倍，存储空间的利用率太低

1. 索引顺序文件中，同样会为文件建立一张索引表
2. 并不是每个记录对应一个索引表项，而是一组记录对应一个索引表项
3. 索引顺序文件是索引文件和顺序文件思想的结合
4. 索引顺序文件的索引项也不需要按关键字顺序排列，可以方便新表项的插入

![](/images/os-notes/4.文件管理-20250518-140257.jpg)

#### 多级索引顺序文件

![](/images/os-notes/4.文件管理-20250518-140505.jpg)

- 为了进一步提高检索效率，可以为顺序文件建立多级索引表
- 要为 N 个记录的文件建立 K 级索引，则最优的分组是每组 $N^{1/(K+1)}$ 个记录
- 检索一个记录的平均查找次数是
$$
\left( \frac{N_1}{K + 1} \div 2 \right) \times (K + 1)
$$

### 文件目录

#### 知识总览

![](/images/os-notes/4.文件管理-20250518-141219.jpg)

#### 文件控制块

1. 目录本身就是一种有结构文件，由一条条记录组成。
2. 每条记录对应一个放在该目录下的文件
3. 目录文件中的一条记录就是一个“文件控制块”

- 文件控制块（FCB）
	1. FCB 实现了文件名和文件之间的映射
	2. 使用户（用户程序）可以实现“按名存取”
	3. FCB 的有序集合称为“文件目录”，一个FCB就是一个文件目录项
	4. FCB 中包含了
		- 文件的基本信息（文件名、物理地址、逻辑结构、物理结构等）
		- 存取控制信息（是否可读/可写、禁止访问的用户名单等）
		- 使用信息（如文件的建立时间、修改时间等）
	5. 最重要，最基本的还是文件名、文件存放的物理地址。

![](/images/os-notes/4.文件管理-20250518-152221.jpg)

- 需要对目录进行哪些操作？ 
	1. 搜索：当用户使用一个文件时，系统根据文件名搜索目录，找到对应的目录项 
	2. 创建文件：创建一个新文件时，需要在其所属的目录中增加一个目录项 
	3. 删除文件：当删除一个文件时，需要在目录中删除相应的目录项 
	4. 显示目录：用户可以请求显示目录的内容，如显示该目录中的所有文件及相应属性 
	5. 修改目录：某些文件属性保存在目录中，因此这些属性变化时要修改相应的目录项

#### 目录结构——单级目录结构

1. 单级目录实现了“按名存取”，但是不允许文件重名
2. 在创建一个文件时，需要先检查目录表中有没有重名文件
3. 确定不重名后才能允许建立文件，并将新文件对应的目录项插入目录表中
4. 显然，单级目录结构不适用于多用户操作系统
5. 整个系统中只有一张目录表

#### 目录结构——两级目录结构

早期的多用户操作系统，采用两级目录结构

- 主文件目录（MFD，MasterFileDirectory）
	1. 主文件目录记录用户名及相应用户文件目录的存放位置
		
- 用户文件目录（UFD，UserFlieDirectory）
	1. 用户文件目录由该用户的文件FCB组成
	2. 允许不同用户的文件重名。文件名虽然相同，但是对应的其实是不同的文件
	3. 两级目录结构允许不同用户的文件重名，也可以在目录上实现实现访问限
	4. 通过检查此时登录的用户名是否匹配来分辨不同文件
	5. 但是两级目录结构依然缺乏灵活性，用户不能对自己的文件进行分类

#### 目录结构——多级目录结构

多级目录结构又称树形目录结构

![](/images/os-notes/4.文件管理-20250518-153036.jpg)

1. 用户（或用户进程）要访问某个文件时用==文件路径名来标识文件==
2. 文件路径名是个字符串，各级目录之间用“/”隔开，从根目录出发的路径称为绝对路径 

> [!note] 例如：自拍.jpg 的绝对路径是“/照片/2015-08/自拍.jpg” 
> 1. 系统根据绝对路径一层一层地找到下一级目录
> 2. 刚开始从外存读入根目录的目录表
> 3. 找到 “照片” 目录的存放位置后，从外存读入对应的目录表
> 4. 再找到 “2015-08” 目录的存放位置，再从外存读入对应目录表
> 5. 最后才找到文件 “自拍.jpg” 的存放位置
> 6. 整个过程需要 3 次读磁盘 I/O 操作

1. 若连续访问同一目录内的多个文件，每次都从根目录开始查找，是很低效的
2. 因此可以设置一个“当前目录”
3. 然后使用从当前目录出发的“相对路径”
4. 引入“当前目录”和“相对路径”后，磁盘 I/O 的次数减少了。提升了访问文件的效率

树形目录结构很方便对文件进行分类，层次结构清晰，能有效地进行文件的管理和保护

#### 目录结构——无环图目录结构

树形目录结构很方便对文件进行分类，层次结构清晰，能有效地进行文件的管理和保护
但是，树形结构不便于实现文件的共享。为此，提出了“无环图目录结构”

![](/images/os-notes/4.文件管理-20250518-153913.jpg)

1. ==让不同文件名指向同一文件==，甚至可以指向同一个目录（共享同一目录下的所有内容）
2. 需要为==每个共享结点设置一个共享计数器==，用于记录此时有多少个地方在共享该结点
3. 用户提出删除结点的请求时
4. 只是删除该用户的 FCB、并使==共享计数器减 1==，并不会直接删除共享结点
5. ==只有共享计数器减为 0 时，才删除结点==。
6. 注意：共享文件不同于复制文件
7. 在共享文件中，由于各用户指向的是同一个文件
8. 因此==只要其中一个用户修改了文件数据，那么所有用户都可以看到文件数据的变化==

#### 索引结点（FCB的改进）

![](/images/os-notes/4.文件管理-20250518-154305.jpg)


1. 当找到文件名对应的目录项时，将索引结点调入内存
2. 索引结点中记录文件的各种信息
3. 包括文件在外存中的存放位置，根据“存放位置”即可找到文件。
4. 存放在外存中的索引结点称为“==磁盘索引结点==”
5. 当索引结点放入内存后称为“==内存索引结点==” 
6. 相比之下内存索引结点中需要增加一些信息，比如：文件是否被修改、有几个进程访问

### 文件的物理结构（文件分配方式）

#### 知识总览

![](/images/os-notes/4.文件管理-20250518-154544.jpg)

![](/images/os-notes/4.文件管理-20250518-154625.jpg)

#### 文件块、磁盘块     

类似于内存分页，磁盘中的存储单元也会被分为一个个“块/磁盘块/物理块”。
很多操作系统中，==磁盘块的大小与内存块、页面的大小相同==

内存与磁盘之间的数据交换（即读/写操作、磁盘I/O）都是以 “块”为单位进行的。
即每次读入一块，或每次写出一块

1. 在内存管理中，进程的逻辑地址空间被分为一个一个页面 
2. 在外存管理中，为方便管理，文件的逻辑地址空间也被分为了一个一个的文件“块”
3. 于是文件的逻辑地址也可以表示为（逻辑块号，块内地址）的形式
4. 用户通过逻辑地址来操作自己的文件
5. ==操作系统负责实现从逻辑地址到物理地址的映射==
6. ==操作系统为文件分配存储空间都是以块为单位的==

#### 文件分配方式——连续分配

==连续分配方式要求每个文件在磁盘上占有一组连续的块==

1. 用户给出要访问的逻辑块号，操作系统找到该文件对应的目录项（FCB）
2. 物理块号 = 起始块号 + 逻辑块号
3. 可以直接算出逻辑块号对应的物理块号
4. 因此==连续分配支持顺序访问和直接访问（即随机访问）==

- 用户通过逻辑地址来操作文件，操作系统如何实现从逻辑地址到物理地址的映射？
	1. （逻辑块号，块内地址） → （物理块号，块内地址）
	2. 只需转换块号就行，块内地址保持不变

![](/images/os-notes/4.文件管理-20250518-155406.jpg)

- 结论
	1. ==连续分配的文件在顺序读/写时速度最快==
	2. 物理上采用连续分配的文件不方便拓展
	3. 物理上采用连续分配， 存储空间利用率低，会产生难以利用的磁盘碎片 
	4. 可以用==紧凑==来处理碎片，需要耗费很大的时间代价

#### 文件分配方式——链接分配

链接分配==采取离散分配的方式==，可以为文件分配离散的磁盘块

分为==隐式链接==和==显式链接==两种

#### 链接分配——隐式链接

==隐式链接==——除文件的最后一个盘块之外，每个盘块中都存有指向下一个盘块的指针

1. 文件目录包括文件第一块的指针和最后一块的指针 
2. 目录中记录了文件存放的起始块号和结束块号
3. 也可以增加一个字段来表示文件的长度 
4. 除了文件的最后一个磁盘块，每个磁盘块中都会保存指向下一个盘块的指针
5. 这些指针对用户是透明的 

- 优点：很方便文件拓展，不会有碎片问题，外存利用率高
- 缺点：只支持顺序访问，不支持随机访问，查找效率低，指向下一个盘块的指针耗费存储空间

![](/images/os-notes/4.文件管理-20250520-175200.jpg)

> 如何实现文件的逻辑块号到物理块号的转变？

1. 用户给出要访问的逻辑块号 i，操作系统找到该文件对应的目录项（FCB）… 	
2. 从目录项中找到起始块号（即 0 号块），将 0 号逻辑块读入内存
3. 由此知道 1 号逻辑块存放的物理块号	
4. 于是读入 1 号逻辑块，再找到 2 号逻辑块的存放位置 ……
5. 因此读入  i 号逻辑块，总共需要 i+1 次磁盘 I/O

- 结论：
	1. ==采用链式分配（隐式链接）方式的文件，只支持顺序访问，不支持随机访问==
	2. 查找效率低，同时指向下一个盘块的指针也需要耗费少量的存储空间

> 是否方便拓展文件？

1. 若此时要拓展文件，可以找一个空闲磁盘块
2. 挂到文件的磁盘块链尾，并修改文件的 FCB

-  结论：
	1.  采用隐式链接的链接分配方式，很方便文件拓展
	2. 另外，所有的空闲磁盘块都可以被利用，不会有碎片问题， 外存利用率高

#### 链接分配——显式链接

==显式链接==把用于链接文件各物理块的指针显式地存放在一张表中，即文件分配表
文件分配表（FAT，File Allocation Table）

![](/images/os-notes/4.文件管理-20250520-193919.jpg)

假设某个新创建的文件 “aaa” 依次存放在磁盘块  2 → 5 → 0 → 1 
假设某个新创建的文件 “bbb” 依次存放在磁盘块 4 → 23 → 3 

- 注意：
	1. 一个磁盘仅设置一张FAT。 
	2. 开机时，将FAT读入内存，并常驻内存。
	3. FAT的各个表项在物理上连续存储，且每一个表项长度相同，因此可以隐含“物理块号”

> 如何实现文件的逻辑块号到物理块号的转变？

1. 用户给出要访问的逻辑块号 i ，操作系统找到该文件对应的目录项（FCB）… 
2. 从目录项中找到起始块号
3. 若 i >0 ，则查询内存中的文件分配表 FAT， 往后找到 i 号逻辑块对应的物理块号
4. ==逻辑块号转换成物理块号的过程不需要读磁盘操作== 

- 结论：
	1. 采用==链式分配（显式链接）==方式的文件，支持顺序访问，也支持随机访问
	2. ==想访问 i 号逻辑块时，并不需要依次访问之前的 0~（i-1） 号逻辑块==
	3. 由于块号转换的过程不需要访问磁盘，因此相比于隐式链接来说，访问速度快很多
	4. 显然，==显式链接也不会产生外部碎片，也可以很方便地对文件进行拓展==

#### 链接分配（总结）

链接分配采取离散分配的方式，可以为文件分配离散的磁盘块 ，分为==隐式链接==和==显式链接==两种

- 考试题目中遇到未指明隐式/显式的“链接分配”，默认指的是隐式链接的链接分配

> 隐式链接——除文件的最后一个盘块之外，每个盘块中都存有指向下一个盘块的指针

1. 文件目录包括文件第一块的指针和最后一块的指针
2. 优点：很方便文件拓展，不会有碎片问题，外存利用率高
3. 缺点：只支持顺序访问，不支持随机访问，查找效率低，指向下一个盘块的指针耗费存储空间

> 显式链接——把用于链接文件各物理块的指针显式地存放在一张表中，即文件分配表

1. 一个磁盘只会建立一张文件分配表
2. 开机时文件分配表放入内存，并常驻内存
3. 优点：
	1. 方便文件拓展，没有碎片问题，外存利用率高，==支持随机访问==
	2. 相比于隐式链接来说，==地址转换时不需要访问磁盘，因此文件的访问效率更高==
4. 缺点：文件分配表的需要占用一定的存储空间

#### 文件分配方式——索引分配

1. 索引分配允许文件离散地分配在各个磁盘块中
2. ==系统会为每个文件建立一张索引表==，索引表中==记录了文件的各个逻辑块对应的物理块==
3. 索引表的功能类似于内存管理中的页表——建立逻辑页面到物理页之间的映射关系
4. 索引表存放的磁盘块称为==索引块==
5. 文件数据存放的磁盘块称为==数据块==

![](/images/os-notes/4.文件管理-20250520-201739.jpg)

假设某个新创建的文件“aaa”的数据依次存放在磁盘块 2 → 5 → 13 → 9

1. 在显式链接的链式分配方式中，文件分配表 FAT 是一个磁盘对应一张
2. 而索引分配方式中，索引表是一个文件对应一张
3. 可以用固定的长度表示物理块号
4. 假设磁盘总容量为 1TB = 240B，磁盘块大小为 1KB，共有230个磁盘块，可用 4B表示磁盘块号
5. 因此，索引表中的“逻辑块号”可以是隐含的

- 如果==一个磁盘块装不下文件的整张索引表==，要如何解决这个问题？
	1. 链接方案	
	2. 多层索引 
	3. 混合索引

#### 索引分配——链接方案

链接方案：如果索引表太大，一个索引块装不下，那么可以将多个索引块链接起来存放。

1. 假设磁盘块大小为 1KB，一个索引表项占 4B，则一个磁盘块只能存放 256 个索引项
2. 若一个文件大小为 256\*256KB = 65,536KB = 64MB 
3. 该文件共有 256\*256 个块，也就对应 256\*256 个索引项
4. 也就需要 256 个索引块来存储，这些索引块用链接方案连起来

![](/images/os-notes/4.文件管理-20250521-103421.jpg)

- 若想要访问文件的最后一个逻辑块， 就必须找到最后一个索引块（第 256 个索引块）
	1. 而各个索引块之间是用指针链接起来的，因此必须先顺序地读入前 255 个索引块 
	2. 这显然是很低效的，如何解决呢？

#### 索引分配——多层索引

多层索引：建立多层索引（原理类似于多级页表），使第一层索引块指向第二层的索引块

1. 若采用多层索引，则各层索引表大小不能超过一个磁盘块
2. 采用 K 层索引结构, 且顶级索引表未调入内存, 则访问一个数据块只需要 K+1 次读磁盘操作
3. 可根据逻辑块号算出应该查找索引表中的哪个表项

![](/images/os-notes/4.文件管理-20250521-104349.jpg)



假设磁盘块大小为 1KB，一个索引表项占 4B，则一个磁盘块只能存放 256 个索引项 

> 若采用两层索引

1. 则该文件的最大长度可以到 256\*256\*1KB = 65,536KB = 64MB 
2. 若要访问 1026 号逻辑块
3. 则 1026/256=4，1026%256=2 
4. 因此可以将一级索引表调入内存，查询 4 号表项
5. 将其对应的二级索引表调入内存，再查询二级索引表的 2 号表项
6. 即可知道 1026 号逻辑块存放的磁盘块号了
7. 访问目标数据块，需要 3 次磁盘 I/O

> 若采用三层索引

1. 则文件的最大长度为 256\*256\*256\*1KB = 16GB 
2. 类似的，访问目标数据块，需要 4 次磁盘 I/O  

#### 索引分配——混合索引

混合索引：多种索引分配方式的结合。

1. 一个文件的顶级索引表中，
2. 既包含直接地址索引（直接指向数据块）
3. 又包含一级间接索引（指向单层索引表）
4. 还包含两级间接索引（指向两层索引表）

![](/images/os-notes/4.文件管理-20250521-115400.jpg)

#### 索引分配（总结）

1. 索引分配允许文件离散地分配在各个磁盘块中，系统会==为每个文件建立一张索引表==
2. 索引表中记录了文件的各个逻辑块对应的物理块
3. 索引表的功能类似于内存管理中的页表——建立逻辑页面到物理页之间的映射关系
4. 索引表存放的磁盘块称为==索引块==
5. 文件数据存放的磁盘块称为==数据块==

- 若文件太大，索引表项太多，可以采取以下三种方法解决： 
	1. ==链接方案==：
		1. 如果索引表太大，一个索引块装不下，那么可以将多个索引块链接起来存放
		2. 缺点：若文件很大，索引表很长，就需要将很多个索引块链接起来
		3. 想要找到 i 号索引块，必须先依次读入 0~(i-1) 号索引块
		4. 这就导致磁盘 I/O 次数过多，查找效率低下 
	2. ==多层索引==：
		1. 建立多层索引（原理类似于多级页表），使第一层索引块指向第二层的索引块
		2. 还可根据文件大小的要求再建立第三层、第四层索引块
		3. 采用 K 层索引结构，顶级索引表未调入内存，访问一个数据块需要 K+1 次操作
		4. 缺点：即使是小文件，访问一个数据块依然需要 K+1 次读磁盘
	3. ==混合索引==：
		1. 多种索引分配方式的结合
		2. 例如，一个文件的顶级索引表中，
		3. 既包含直接地址索引（直接指向数据块）
		4. 又包含一级间接索引（指向单层索引表）
		5. 还包含两级间接索引（指向两层索引表）
		6. 优点：对于小文件来说，访问一个数据块所需的读磁盘次数更少。 

> 总结：

- ==要能自己分析访问某个数据块所需要的读磁盘次数==
- ==要会根据多层索引、混合索引的结构计算出文件的最大长度==

1. 各级索引表最大不能超过一个块
2. FCB 中会存有指向顶级索引块的指针，因此可以根据 FCB 读入顶级索引块
3. 每次读入下一级的索引块都需要一次读磁盘操作
4. 另外，要注意题目条件——顶级索引块是否已调入内存

> 支持随机访问

- “文件的某种逻辑结构支持随机存取/随机访问”是指：
	1. 采用这种逻辑结构的文件
	2. 可以根据记录号直接算出该记录对应的逻辑地址（逻辑块号，块内地址）

1. “==定长记录的顺序文件支持随机访问==”——可以直接算出 i 号记录对应的逻辑地址
2. ==若文件记录不定长，只能先顺序遍历==前 i-1 个记录，因此记录不定长时不支持随机访问

### 文件存储空间管理

#### 知识总览

#### 存储空间的划分与初始化

存储空间的划分：将物理磁盘划分为一个个文件卷(逻辑卷、逻辑盘)

![](/images/os-notes/4.文件管理-20250528-132008.jpg)

#### 空闲表法

为一个磁盘创建一个表，来存储空闲磁盘块的位置

![](/images/os-notes/4.文件管理-20250528-132443.jpg)

- 如何分配磁盘块：
	1. 与内存管理中的动态分区分配很类似，为一个文件分配连续的存储空间
	2. 同样可采用首次适应、最佳适应、最坏适应等算法来决定要为文件分配哪个区间

- 如何回收磁盘块：
	1. 与内存管理中的动态分区分配很类似，回收某个存储区时有四种情况
	2. 回收区的前后都没有相邻空闲区
	3. 回收区的前后都是空闲区
	4. 回收区前面是空闲区
	5. 回收区后面是空闲区
	6. 总之，回收时需要注意表项的合并问题

#### 空闲链表法

空闲链表法分为空闲盘块链和空闲盘区链

![](/images/os-notes/4.文件管理-20250528-132740.jpg)

#### 空闲盘块链

1. 操作系统保存着链头、链尾指针
2. 分配：若某文件申请 K 个盘块，从链头开始依次摘下 K 个盘块分配，并修改空闲链的链头指针
3. 回收：回收的盘块依次挂到链尾，并修改空闲链的链尾指针
4. 适用于离散分配的物理结构
5. 为文件分配多个盘块时可能要重复多次操作

![](/images/os-notes/4.文件管理-20250528-132910.jpg)

#### 空闲盘区链

1. 操作系统保存着链头、链尾指针
2. 若某文件申请 K 个盘块，则可以采用首次适应、最佳适应等算法
3. 从链头开始检索，按照算法规则找到一个大小符合要求的空闲盘区，分配给文件
4. 若没有合适的连续空闲块，也可以将不同盘区的盘块同时分配给一个文件
5. 注意分配后可能要修改相应的链指针、盘区大小等数据
6. 若回收区和某个空闲盘区相邻，则需要将回收区合并到空闲盘区中
7. 若回收区没有和任何空闲区相邻，将回收区作为单独的一个空闲盘区挂到链尾
8. 离散分配、连续分配都适用
9. 为一个文件分配多个盘块时效率更高

![](/images/os-notes/4.文件管理-20250528-133120.jpg)

#### 位示图法

1. 位示图：每个二进制位对应一个盘块。
2. 位示图法是一种用于资源管理的数据结构，主要用于磁盘存储管理或内存分配
3. 通过二进制位（0/1）表示资源的使用状态
4. 核心思想是用每一位（bit）代表一个存储块或资源单元，标记其是否被占用

$$
\text{位示图大小} = \left\lceil \frac{N}{字长} \right\rceil \text{ 字节}
$$

#### 成组链接法

1. 空闲表法、空闲链表法不适用于大型文件系统，因为空闲表或空闲链表可能过大
2. UNIX 系统中采用了成组链接法对磁盘空闲块进行管理
3. 文件卷的目录区中专门用一个磁盘块作为“超级块”
4. 当系统启动时需要将超级块读入内存
5. 要保证内存与外存中的“超级块”数据一致

![](/images/os-notes/4.文件管理-20250528-135315.jpg)

---

![](/images/os-notes/4.文件管理-20250528-135419.jpg)


- Eg : 需要 100 个空闲块
	1. 检查第一个分组的块数是否足够。100=100，是足够的。
	2. 分配第一个分组中的 100 个空闲块。
	3. 由于 300 号块内存放了再下一组的信息
	4. 因此 300 号块的数据需要复制到超级块

- Eg : 需要 1 个空闲块
	1. 检查第一个分组的块数是否足够。1<100，因此是足够的。
	2. 分配第一个分组中的 1 个空闲块，并修改相应数据

### 文件的基本操作

#### 知识总览

#### 创建文件

创建文件：点击新建后，图形化交互进程在背后调用了 Create 系统调用

- 进行 Create 系统调用时，需要提供的几个主要参数：
	1. 所需的外存空间大小
	2. 文件存放路径
	3. 文件名

- 操作系统在处理 Create 系统调用时，主要做了两件事：
	1. ==在外存中找到文件所需的空间==，结合空闲链表法、位示图、成组链接法等管理策略，找到空闲空间
	2. 根据文件存放路径的信息找到该目录对应的录文件，==在目录中创建该文件对应的目录项==。目录项中包含了文件名、文件在外存中的存放位置等信息。

#### 删除文件

删除文件：图形化交互进程通过操作系统提供的 delete 系统调用，将文件数据从外存中删除

- 进行 Delete 系统调用时，需要提供的几个主要参数：
	1. 文件存放路径
	2. 文件名

- 操作系统在处理 Delete 系统调用时，主要做了几件事：
	1. 根据文件存放路径找到相应的目录文件，从目录中==找到文件名对应的目录项==
	2. 根据该目录项记录的文件在外存的存放位置、文件大小等信息，==回收文件占用的磁盘块==
	3. 从目录表中删除文件对应的目录项

#### 打开文件

如果所需文件已在内存中打开，再操作文件就不需要每次都重新查目录，可以加快文件的访问速度

- 在用户使用 open 系统调用“打开文件”，需要提供的几个主要参数：
	1. 文件存放路径
	2. 文件名
	3. 要对文件的操作类型（如：r 只读，rw 读写等）

- 操作系统在处理 open 系统调用时，主要做了几件事：

	1. 根据文件存放路径找到相应的目录文件，从目录==中找到文件名对应的的目录项==
	2. 并检查该用户是否有指定的操作权限
	3. ==将目录项复制到内存中的“打开文件表”中==，并将对应表目的编号返回给用户
	4. 之后==用户使用打开文件表的编号来指明要操作的文件==

---

![](/images/os-notes/4.文件管理-20250521-155647.jpg)

- 读写指针：记录了该进程对文件的读/写操作进行到的位置
- 访问权限：控制用户对文件的操作
- 打开计数器：记录此时有多少个进程打开了此文件

1. 系统的打开文件表可以方便实现某些文件管理的功能
2. 例如在 windows 系统中，我们尝试删除某个 txt 文件
3. 如果此时该文件已被某个“记事本进程打开，则系统会提示我们“暂时无法删除该文件”
4. 其实系统在背后做的事就是先检查了系统打开文件表，确认此时是否有进程正在使用该文件

- 进程使用完文件后，要“关闭文件“操作系统在处理 Close 系统调用时，主要做了几件事：
	1. 将进程的打开文件表相应表项删除
	2. 回收分配给该文件的内存空间等资源
	3. 系统打开文件表的打开计数器 count 减 1，若 count = 0，则删除对应表项

#### 读文件

读文件：通过 read 系统调用，将文件数据读入内存，让 CPU 处理 

- 进程使用 read 系统调用完成写操作
	1. 需要指明是哪个文件
	2. 还需要指明要读入多少数据
	3. 指明读入的数据在内存中的存放位置

#### 写文件

写文件：通过 write 系统调用，将更改过的文件数据写回外存

- 进程使用 write 系统调用完成写操作
	1. 需要指明是哪个文件 （有时可以提供文件在打开文件表中的索引号）
	2. 需要指明要写出多少数据
	3. 写回外存的数据在内存中的存放位置

### 文件共享

#### 知识总览

![](/images/os-notes/4.文件管理-20250521-191020.jpg)

1. 多个用户共享同一个文件，意味着系统中只有“一份”文件数据
2. 并且只要某个用户修改了该文件的数据，其他用户也可以看到文件数据的变化
3. 如果是多个用户都“复制”了同一个文件，那么系统中会有“好几份”文件数据
4. 其中一个用户修改了自己的那份文件数据，对其他用户的文件数据并没有影响

#### 基于索引结点的共享方式（硬链接）

知识回顾：索引结点，是一种文件目录瘦身策略。由于检索文件时只需用到文件名，因此可以将除了文件名之外的其他信息放到索引结点中。这样目录项就只需要包含文件名、索引结点指针。

![](/images/os-notes/4.文件管理-20250521-192048.jpg)


- 索引结点中设置一个链接计数变量 count ，用于表示链接到本索引结点上的用户目录项数
	1. 若 count=2，说明此时有两个用户目录项链接到该索引结点上
	2. 或者说是有两个用户在共享此文件。
	3. 若某个用户决定“删除”该文件，则只是要把用户目录中与该文件对应的目录项删除
	4. 此时索引结点的 count 值减 1
	5. 若 count>0，说明还有用户要使用该文件，暂时不能删除文件数据，否则会导致指针悬空
	6. 当 count=0 时系统负责删除文件

#### 口令保护

为文件设置一个“口令”，用户请求访问该文件时必须提供“口令”

1. 口令一般存放在文件对应的 FCB 或索引结点中
2. 用户访问文件前需要先输入“口令”

优点：保密性强，不需要在系统中存储“密码“
缺点：编码/译码，或者说加密/解密要花费一定时间

### 文件层次机构

![](/images/os-notes/4.文件管理-20250528-142542.jpg)

用一个例子来辅助记忆文件系统的层次结构：

1. 假设某用户请求删除文件“D:/工作目录/学生信息.xlsx”的最后100条记录
2. 用户需要通过操作系统提供的接口发出上述请求一用户接口
3. 由于用户提供的是文件的存放路径
4. 因此需要操作系统一层一层地查找目录，找到对应的目录项――文件目录系统
5. 不同的用户对文件有不同的操作权限，因此为了保证安全，需要检查用户是否有访问权限―一存取控制模块（存取控制验证层)
6. 验证了用户的访问权限之后，需要把用户提供的“记录号”转变为对应的逻辑地址――逻辑文件系统与文件信息缓冲区
7. 知道了目标记录对应的逻辑地址后，还需要转换成实际的物理地址――物理文件系统
8. 要删除这条记录，必定要对磁盘设备发出请求――设备管理程序模块
9. 删除这些记录后，会有一些盘块空闲，因此要将这些空闲盘块回收――辅助分配模块

### 磁盘的结构

#### 磁盘、磁道、扇区

![](/images/os-notes/4.文件管理-20250522-204821.jpg)

1. 磁盘的盘面被划分成一个个磁道。这样的一个“圈”就是一个磁道
2. 磁盘的表面由一些磁性物质组成，可以用这些磁性物质来记录二进制数据
3. 最内侧磁道上的扇区面积最小，因此数据密度最大
4. 一个磁道又被划分成一个个扇区，每个扇区就是一个“磁盘块”
5. 各个扇区存放的数据量相同（如 1 KB）

#### 如何在磁盘中读/写数据

1. 需要把“磁头”移动到想要读/写的扇区所在的磁道
2. 磁盘会转起来，让目标扇区从磁头下面划过，才能完成对扇区的读/写操作

#### 盘面、柱面

![](/images/os-notes/4.文件管理-20250522-205233.jpg)

1. 一个盘片可能会有两个盘面
2. 每个盘面对应一个磁头
3. 所有的磁头都是连在同一个磁臂上的，因此所有磁头只能“共进退”
4. ==所有盘面中相对位置相同的磁道组成柱面==

#### 磁盘的物理地址

可用（柱面号，盘面号，扇区号）来定位任意一个“磁盘块”

- 可根据地址读取一个“块”
	1. 根据“柱面号”移动磁臂，让磁头指向指定柱面
	2. 激活指定盘面对应的磁头
	3. 磁盘旋转的过程中，指定的扇区会从磁头下面划过，这样就完成了对指定扇区的读/写

#### 磁盘的分类

- 活动头磁盘：磁头可以移动，磁臂可以来回伸缩来带动磁头定位磁道

![](/images/os-notes/4.文件管理-20250522-205726.jpg)

- 固定头磁盘：磁头不可移动，磁盘中每个磁道有一个磁头

![](/images/os-notes/4.文件管理-20250522-205649.jpg)

- 可换盘磁盘：可以更换盘片
- 固定盘磁盘：不可更换盘片

### 磁盘调度算法

#### 知识总览

![](/images/os-notes/4.文件管理-20250523-084659.jpg)

#### 一次磁盘读/写操作需要的时间

> ==寻找时间（寻道时间）==T

1. 在读/写数据前，将磁头移动到指定磁道所花时间	
2. ==启动磁头臂==是需要时间的，假设耗时为 s
3. ==移动磁头==也是需要时间的
4. 假设磁头匀速移动，每跨越个磁道耗时为 m，总共需要跨越 n 条磁道
5. 则：寻道时间 T<sub>s</sub> = s + m\*n
6. 现在的硬盘移动一个磁道大约需要 0.2 ms，磁臂启动时间约为 2 ms

> ==延迟时间 T<sub>R</sub>== 

1. 通过旋转磁盘，使磁头定位到目标扇区所需要的时间	
2. 设磁盘转速为 r ，则平均所需的延迟时间 T = (1/2)\*(1/r) = 1/2 r
3. 硬盘的典型转速为 5400 转/分，或 7200 转/分
4. 1/r 就是磁盘转一圈需要的时间
5. 找到目标扇区平均需要转半圈，因此再乘以 1/2 

> 传输时间 T

1. 从磁盘读出或向磁盘写入数据所经历的时间
2. 假设磁盘转速为 r，此次读/写的字节数为 b，每个磁道上的字节数为 N
3. 传输时间 T = (1/r)\*(b/N) = b/(rN)
4. 每个磁道要可存 N 字节的数据，因此 b 字节的数据需要 b/N 个磁道才能存储
5. 而读/写一个磁道所需的时间刚好又是转一圈所需要的时间 1/r

---

==总的平均存取时间==：T<sub>a</sub> = T<sub>s</sub> + 1/2 r + b/(rN)

1. 操作系统的磁盘调度算法会直接影响寻道时间
2. 延迟时间和传输时间都与磁盘转速相关，且为线性相关
3. 而转速是硬件的固有属性，因此操作系统也无法优化延迟时间和传输时间

#### 磁盘调度算法

>先来先服务算法（FCFS）

FCFS 算法会根据进程请求访问磁盘的==先后顺序==进行调度

$$
\frac{\text{磁头移动距离}}{\text{进程数}} = \text{平均寻找长度}
$$

- 优点：公平；如果请求访问的磁道比较集中，算法性能还可以
- 缺点：如果请求访问的磁道很分散，则 FCFS 在性能上很差，寻道时间长

> 最短寻找时间优先 (SSTF)

SSTF 算法会优先处理与==当前磁头最近==的磁道

![](/images/os-notes/4.文件管理-20250523-095151.jpg)

1. 可以保证每次的寻道时间最短，但是并不能保证总的寻道时间最短
2. 其实就是贪心算法的思想，只是选择眼前最优，但是总体未必最优
3. 产生饥饿的原因：磁头在一个小区域内来回来去地移动

- 优点：性能较好，平均寻道时间短
- 缺点：可能产生“==饥饿==”现象 

> 扫描算法(SCAN)

SCAN 算法只允许磁头移动到最边缘磁道的时候才能改变方向

![](/images/os-notes/4.文件管理-20250523-095046.jpg)

1. 磁头移动到最外侧磁道的时候才能往内移动
2. 磁头移动到最内侧磁道的时候才能往外移动
3. 由于磁头移动的方式很像电梯，因此也叫==电梯算法==

- 优点：性能较好，平均寻道时间较短，不会产生饥饿现象
- 缺点：
	1. 只有到达最边上的磁道时才能改变磁头移动方向，可能造成性能浪费
	2. SCAN 算法对于各个位置磁道的响应频率不平均

> LOOK 调度算法

LOOK 度算法在磁头移动方向上已经没有别的请求时，可以立即改变磁头移动方向

![](/images/os-notes/4.文件管理-20250523-100011.jpg)

避免扫描算法（SCAN）为使磁头移动到最边磁道而造成的性能浪费

- 优点：磁头不需要每次都移动到最边缘磁道再改变方向，寻道时间进一步缩短

>循环扫描算法 (C-SCAN)

C-SCAN 算法让磁头移动到最边缘磁道再改变方向，直接回到起始端而不处理任何请求

![](/images/os-notes/4.文件管理-20250523-144235.jpg)

避免 SCAN 算法对于各个位置磁道的响应频率不平均

- 优点：比起 SCAN 来，对于各个位置磁道的响应频率很平均
- 缺点：
	1. 只有到达最边上的磁道时才能改变磁头移动方向，造成性能浪费
	2. 磁头返回时必须返回到另一端的边缘磁道，造成性能浪费
	3. 比起 SCAN 算法来，平均寻道时间更长

> C-LOOK 调度算法

C-LOOK 算法在磁头移动方向上已经没有别的请求时，可以立即让磁头返回，并且磁头只需要返回到最靠近边缘的、并且需要访问的磁道上即可

![](/images/os-notes/4.文件管理-20250523-145048.jpg)

- 优点: 比起 C-SCAN 算法来，不需要每次都移动到最外侧或最内侧才改变磁头方向，使寻道时间进一步缩短

### 减少延迟时间

2. 可以使读取连续的逻辑扇区所需要的延迟时间更小

#### 磁盘地址结构的设计

- 为什么磁盘的物理地址是（柱面号, 盘面号, 扇区号），而不是（盘面号, 柱面, 扇号区号）
- 读取地址连续的磁盘块时采用 (柱面号,盘面号, 扇区号) 的地址结构可以减少磁头移动消耗的时间

> [!note] 假设某磁盘有 8 个柱面/磁道（假设最内侧柱面/磁道号为 0）4 个盘面, 8 个扇区
> 
> 则可用 3 个二进制位表示柱面，2 个二进制位表示盘面，3 个二进制位表示扇区
> 
> - 若物理地址结构是（盘面号, 柱面号, 扇区号）
> 	1. 需要连续读取物理地址 (00,000,000) ~ (00,001,111) 的扇区
> 	2. (00, 000, 000) ~ ( 00, 000, 111) 转两圈可读完
> 	3. 再读取物理地址相邻的区域
> 	4. 即 (00, 001, 000) ~ ( 00, 001, 111)==需要启动磁头臂，将磁头移动到下一个磁道==
> 
> - 若物理地址结构是（柱面号, 盘面号, 扇区号）
> 	1. 需要连续读取物理地址 (000,00,000)~(000,01,111) 的扇区
> 	2. (000, 00, 000) ~ ( 000, 00, 111) 由盘面 0 的磁头读入数据
> 	3. 再读取物理地址相邻的区域，即(000, 01, 000) ~( 000, 01, 111)
> 	4. 由于柱面号/磁道号相同，只是盘面号不同
> 	5. 因此==不需要移动磁头臂，只需要激活相邻盘面的磁头即可==

### 磁盘的管理

#### 磁盘初始化

1. 进行低级格式化（物理格式化），将磁盘的各个磁道划分为扇区
2. 一个扇区通常可分为头、数据区域、尾三个部分组成
3. 管理扇区所需要的各种数据结构一般存放在头、尾两个部分，包括扇区校验码
4. 校验码用于校验扇区中的数据是否发生错误，包括奇偶校验、CRC循环冗余校验码等
5. 将磁盘分区，每个分区由若干柱面（磁道）组成（即分为我们熟悉的C盘、D盘、E盘）
6. 进行逻辑格式化，创建文件系统。
7. 包括创建文件系统的根目录、初始化存储空间管理所用的数据结构（如位示图）

![](/images/os-notes/4.文件管理-20250528-144757.jpg)

#### 引导块

1. 计算机开机时需要进行一系列初始化的工作
2. 这些初始化工作是通过执行初始化程序（自举程序）完成的
3. 初始化程序可以放在 ROM （只读存储器）中
4. ROM 中的数据在出厂时就写入了，并且以后不能再修改 

- 无法更新自举程序，因为ROM中的数据无法更改，如何解决呢？
	1. ROM 中只存放很小的“自举装入程序”
	2. 开机时计算机先运行“自举装入程序”，通过执行该程序就可找到引导块
	3. 引导块将完整的“自举程序”读入内存，完成初始化
	4. 完整的自举程序放在磁盘的启动块（即引导块/启动分区）上
	5. 启动块位于磁盘的固定位置
	6. 拥有启动分区的磁盘称为启动磁盘或系统磁盘（C盘）
    
#### 坏块的管理

1. 坏了、无法正常使用的扇区就是“坏块”。
2. 这属于硬件故障，操作系统是无法修复的
3. 应该将坏块标记出来，以免错误地使用到它

- 对于简单的磁盘
	1. 可以在逻辑格式化时（建立文件系统时）对整个磁盘进行坏块检查
	2. 标明哪些扇区是坏扇区，比如：在FAT表上标明
	3. 在这种方式中，坏块对操作系统不透明  
      
- 对于复杂的磁盘，磁盘控制器（磁盘设备内部的一个硬件部件）会维护一个坏块链表 
	1. 在磁盘出厂前进行低级格式化（物理格式化）时就将坏块链进行初始化
	2. 会保留一些“备用扇区”，用于替换坏块。这种方案称为扇区备用
	3. 这种处理方式中，坏块对操作系统透明
