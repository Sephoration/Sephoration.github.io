### 内存的基础知识

#### 知识总览

![](/images/os-notes/3.内存管理-20250519-162400.jpg)

#### 内存的定义和作用

1. 内存是存放数据的硬件
2. 程序执行前==需要先放到内存中才能被 CPU 处理==
3. 缓和 CPU 与硬盘之间的速度矛盾 
4. 通过给==内存的存储单元编地址==可以区分各个程序的数据，保证其并发执行

- 常用的数量单位
	2<sup>10</sup>=1 K  （千） 
	2<sup>20</sup>=1 M（兆，百万） 
	2<sup>30</sup>=1 G  （十亿，千兆）
	
#### 内存地址

1. 内存地址从 0 开始，每个地址对应一个存储单元
2. 将内存分成多个 “存储单元”
3. 如果计算机 “==按字节编址==” ，每个存储单元大小为 1 字节，即 1B（ 8个二进制位）
4. 如果字长为16位的计算机 “==按字编址==” ，存储单元大小为 1 个字（ 16个二进制位）

#### 指令的工作原理

1. 指令的工作基于 “地址” ，每个地址对应一个数据的存储单元
2. 有的系统中，寄存器和内存可能统一编址
3. C 语言程序经过编译、 链接处理后，生成装入模块，即可执行文件
4. 程序经过编译、链接后生成的指令中指明的是逻辑地址（相对地址）

![](/images/os-notes/3.内存管理-20250510-183632.jpg)

#### 逻辑地址 VS 物理地址

逻辑地址：相对地址 
物理地址：绝对地址

编译时产生的指令只关心“相对地址”，实际放入内存中时再想办法根据起始地址得到“绝对地址”

- 如何将指令中的逻辑地址转换为物理地址？

- 策略：三种装入方式
	1. 绝对装入
	2. 静态重定位（可重定位装入） 
	3. 动态重定位（动态运行时装入）

![](/images/os-notes/3.内存管理-20250514-172646.jpg)

#### 装入方式 -- 绝对装入

1. 在编译时，编译程序产生绝对地址的目标代码
2. 编译、链接后得到的装入模块的指令==直接使用绝对地址==
3. 装入程序按照装入模块中的地址，将程序和数据装入内存
4. 绝对装入只适用于==单道程序环境==

#### 装入方式 -- 静态重定位

静态重定位：又称可重定位装入

1. 编译、链接后的装入模块的地址都是从 0 开始的
2. 指令中使用的地址、数据存放的地址都是==相对于起始地址而言的逻辑地址==
3. 可根据内存的当前情况，将装入模块装入到内存的适当位置
4. 装入时对地址进行“==重定位==”，将逻辑地址变换为物理地址

- 特点：
	1. 一个作业装入内存前，必须==分配其要求的全部内存空间==
	2. 作业一旦进入内存后，在运行期间就==不能再移动==，也不能再申请内存空间

#### 装入方式 -- 动态重定位

动态重定位：又称动态运行时装入

1. 编译、链接后的装入模块的地址都从 0 开始
2. 装入程序把装入模块装入内存后，并不会立即把逻辑地址转换为物理地址
3. 而是==把地址转换推迟到程序真正要执行时==才进行
4. 因此装入内存后所有的地址依然是逻辑地址，依赖一个==重定位寄存器==
5. 采用动态重定位时==允许程序在内存中发生移动==

![](/images/os-notes/3.内存管理-20250510-191614.jpg)

#### 编译、链接、装入（装载）

![](/images/os-notes/3.内存管理-20250511-002351.jpg)

1. 编译：将高级语言翻译成机器语言，把用户源代码编译成若干个目标模块
2. 链接：将编译后形成的一组目标模块和所需库函数链接，形成完整装入模块 
3. 装入（装载）：由装入程序将装入模块装入内存运行

#### 链接的三种方式       
         
1. 静态链接：在程序运行前，将各目标模块及所需的库函数连接成一个完整的可执行文件（装入模块），之后不再拆开
2. 装入时动态链接：将各目标模块装入内存时，采取边装入边链接的方式
3. 运行时动态链接：在程序执行中需要该目标模块时，才对它进行链接。其优点是便于修改和更新，便于实现对目标模块的共享

### 内存管理的概念

#### 知识总览

![](/images/os-notes/3.内存管理-20250514-183405.jpg)

1. 操作系统负责==内存空间的分配与回收==
2. 操作系统需要提供某种技术（虚拟技术）从==逻辑上对内存空间进行扩充==
3. 操作系统需要提供==地址转换==功能，负责程序的逻辑地址与物理地址的转换
4. 操作系统需要提供==内存保护==功能，保证各进程在各自的存储空间内运行

#### 地址转换

1. 可使得编程更方便，程序员写程序时只需关注指令、数据的逻辑地址
2. 而逻辑地址到物理地址的转换（==地址重定位==）由操作系统负责	
3. 程序员写程序时不再需要关注物理内存的实际情况

#### 内存保护

1. 方法一：在 CPU 中==设置一对上、下限寄存器==，存放进程的上、下限地址；进程的指令要访问某个地址时，CPU 检查是否越界。
2. 方法二：使用==重定位寄存器==/基址寄存器、==界地址寄存器==/限长寄存器进行越界检查。重定位寄存器中存放的是进程的起始物理地址；界地址寄存器中存放的是进程的最大逻辑地址。

### 覆盖与交换

#### 知识总览

![](/images/os-notes/3.内存管理-20250514-181502.jpg)

#### 覆盖技术

用来解决 “==程序大小超过物理内存总和==” 的问题

- 核心思想：
	1. 将程序分为段/模块，常用的段常驻内存，不常用的在需要时再调入
	2. 内存中分为一个“==固定区==”和若干个“==覆盖区==“
	3. 常用的段放在“固定区”，这些段常驻内存，==调入后就不再调出==
	4. 不常用的放在“覆盖区”，需要时调入内存，==不需要时调出内存==

![](/images/os-notes/3.内存管理-20250512-003709.jpg)

- 覆盖结构==必须由程序员声明==，操作系统完成自动覆盖
- 缺点：==对用户不透明==，增加了用户编程负担 
- 覆盖技术只用于早期的操作系统中

#### 交换技术

设计思想：内存空间紧张时，系统将内存中某些进程暂时==换出==外存，把外存中某些已具备运行条件的进程==换入==内存（进程在内存与磁盘间动态调度）

中级调度（内存调度），决定将哪个处于挂起状态的进程重新调入内存

1. 挂起状态（挂起态，suspend）：暂时换出外存等待的进程状态
2. 挂起状态可以分为就绪挂起、阻塞挂起两种状态

![](/images/os-notes/3.内存管理-20250512-004304.jpg)

1. 具有对换功能的操作系统中，通常把磁盘空间分为==文件区==和==对换区==两部分
2. ==文件区用于存放文件==，为追求利用率，==采用离散分配方式==管理文件区空间
3. 对换区空间只占了磁盘空间的小部分，==被换出的进程数据就存放在对换区==
4. 因对换的速度直接影响系统整体速度，通常对换区==采用连续分配方式==
5. 对换区的 I/O 速度比文件区的更快
6. 交换通常在许多进程运行且内存吃紧时进行，而系统负荷降低就暂停
7. 可优先换出阻塞进程，可换出优先级低的进程
8. 为防止优先级低的进程在被调入内存后很快又被换出，有时考虑驻留时间
9. ==PCB 会常驻内存，不会被换出外存==

#### 知识总结

![](/images/os-notes/3.内存管理-20250512-005520.jpg)

### 连续分配管理方式

#### 知识总览

![](/images/os-notes/3.内存管理-20250512-085824.jpg)

==连续分配==：指为用户进程分配的必须是一个==连续的内存空间==

####  单一连续分配

1. 在单一连续分配方式中，内存被分为==系统区==和==用户区==
2. 系统区通常位于内存的低地址部分，用于存放操作系统相关数据
3. 用户区用于存放用户进程相关数据 
4. ==内存中只能有一道用户程序==，用户程序独占整个用户区空间 
5. 优点：实现简单，==无外部碎片==，可采用覆盖技术扩充内存，不一定需要内存保护
6. 缺点：只能用于单用户、单任务的操作系统中，==有内部碎片==，存储器利用率低
7. 分配给某进程的内存区域中，如果有些部分没有用上，就是“==内部碎片==”

![](/images/os-notes/3.内存管理-20250514-181758.jpg)

#### 固定分区分配

1. 为了在内存中装入多道程序，且程序间不会相互干扰
2. 将整个用户空间==划分为若干个固定大小/不固定大小的分区==
3. 在==每个分区中只装入一道作业==
4. 形成了一种可运行多道程序的内存管理方式

- 分区大小相等：缺乏灵活性，==适合用于用一台计算机控制多个相同对象的场合==
- 分区大小不等：增加了灵活性，满足不同大小的进程需求，根据作业大小划分

==分区说明表== -- 操作系统需要建立一个数据结构来记录分区 

1. 实现各个分区的分配与回收
2. 每个表项对应一个分区，通常按分区大小排列
3. 每表项包括对应分区的 “==大小==、==起始地址==、==状态==”（是否已分配）

![](/images/os-notes/3.内存管理-20250512-091115.jpg)

- 优点：实现简单，==无外部碎片==。 
- 缺点：
	1. 当用户程序太大时，可能所有的分区都不能满足需求
	2. 当用户程序太大时，只能采用覆盖技术，会减低性能
	3. ==会产生内部碎片==，内存利用率低

#### 动态分区分配

动态分区分配又称为可变分区分配

1. 动态分区分配不会预先划分内存分区，而是==根据进程的大小动态地建立分区==
2. 因此系统分区的大小和数目是可变的

把一个新作业装入内存时，须按照一定的==动态分区分配算法==
从空闲分区表（或空闲分区链）中选出一个分区分配给该作业

1. 动态分区分配==没有内部碎片==，但是==有外部碎片==
2. 内部碎片：分配给某进程的内存区域中，有些部分没有用上 
3. 外部碎片：内存中的某些空闲分区由于太小而难以利用
4. 如果内存中空闲空间的总和本可以满足某进程的要求，但进程需要的是一整块连续的内存空间，而 “碎片”不能满足其进程的需求，可以通过==紧凑==（拼凑，Compaction）技术来解决外部碎片。

系统用 “==空闲分区表==” 、“==空闲分区链==” 两种数据结构来记录内存的使用情况

![](/images/os-notes/3.内存管理-20250514-182242.jpg)

- 如何回收多余空间：将相邻的空闲区域合并为一个

### 动态分区分配算法

#### 知识总览

在动态分区分配方式中，多个空闲分区满足需求时，应选哪个分区进行分配 ？

![](/images/os-notes/3.内存管理-20250515-003655.jpg)

- 动态分区分配算法是连续分配管理方式的一种

#### 首次适应算法 (First Fit)

算法思想：每次都从低地址开始查找，找到第一个能满足大小的空闲分区

1. ==空闲分区以地址递增的次序排列==
2. 每次分配内存时顺序查找空闲分区链（或空闲分区表）
3. 找到大小能满足要求的第一个空闲分区

- 特点：
	1. 首次适应算法每次都要从头查找，每次都需要检索低地址的小分区 
	2. 当低地址部分有更小的分区可以满足需求时，更可能用到低地址部分的小分区
	3. 更可能把高地址部分的大分区保留下来（最佳适应算法的优点） 

#### 最佳适应算法 (Best Fit)

算法思想：由于动态分区分配是一种连续分配方式，各进程分配的空间必须是连续的一整片区域；为保证当 “大进程” 到来时能有连续的大片空间，需要尽可能多地留下大片的空闲区， 即==优先使用更小的空闲区==

1. 空闲分区==按容量递增次序链接==
2. 每次分配内存时顺序查找空闲分区链（或空闲分区表）
3. 找到大小能满足要求的第一个空闲分区
 
- 缺点：因选最小分区进行分配，每次留下极小的内存块，==会产生很多外部碎片==

#### 最坏适应算法 (Worst Fit)

又称最大适应算法（Largest Fit）

算法思想：为了解决最佳适应算法的问题 -- 留下太多难以利用的小碎片，可以在每次分配时==优先使用最大的连续空闲区==，这样分配后剩余的空闲区就不会太小

1. 空闲分区==按容量递减次序链接==
2. 每次分配内存时顺序查找空闲分区链（或空闲分区表）
3. 找到大小能满足要求的第一个空闲分区

- 缺点：因选最大分区进行分配，可能导致后续的 “大进程” 没有内存分区可用

#### 邻近适应算法 (Next Fit)

算法思想：首次适应算法每次都从链头开始查找的，可能导致低地址部分出现很多小的空闲分区；而每次分配查找时，都要经过这些分区，增加查找的开销。为解决该问题，可以设置每次从上次查找结束的位置再开始检索

1. 空闲分区以地址递增的顺序排列（可排成一个循环链表）
2. 每次分配内存时==从上次查找结束的位置开始==查找空闲分区链（或空闲分区表）
3. 找到大小能满足要求的第一个空闲分区。

![](/images/os-notes/3.内存管理-20250512-100406.jpg)

- 特点
	1. 可能使得低地址、高地址部分的空闲分区有相同的概率被使用
	2. 导致了高地址部分的大分区更可能被使用，划分为小分区
	3. 可能最后导致无大分区可用（最大适应算法的缺点） 

#### 知识总结

![](/images/os-notes/3.内存管理-20250512-100505.jpg)

综合来看，==四种算法中，首次适应算法的效果反而更好== 

### 基本分页存储管理的基本概念

#### 知识总览

![](/images/os-notes/3.内存管理-20250512-101649.jpg)

连续的分配：为用户进程分配的必须是一个==连续==的内存空间。 
非连续分配：为用户进程分配的可以是一些==分散==的内存空间。

#### 地址空间

![](/images/os-notes/3.内存管理-20250512-101920.jpg)

#### 分页存储

基本思想：把内存分为一个个相等的小分区，再按照分区大小把进程拆分为一个个部分

1. 将内存空间分为一个个==大小相等的分区==（比如：每个分区 4KB）
2. 每个分区就是一个“==页框==” （ 页帧/内存块/物理块/物理页面 ）
3. 每个页框有一个编号，叫 “==页框号==” （页帧号/内存块号/物理块号/物理页号）

- 进程的分页
	1. ==进程的逻辑地址空间==也分为与页框大小相等的一个个部分
	2. 每部分称为一个“==页==”或“==页面==”
	3. 每个页面也有一个编号， 即“页号”，页号也是从0开始

1. 操作系统==以页框为单位为各个进程分配==内存空间
2. 进程的每个页面分别放入一个页框中，进程的页面与内存的页框==一一对应==
3. 各个页面不必连续存放，可以放到不相邻的各个页框中 
4. 进程的最后一个页面可能没有一个页框那么大
5. ==分页存储可能产生内部碎片==，因此页框不能太大，避免产生过大内部碎片造成浪费

#### 页表

为能==知道进程的每个页面在内存中存放的位置==，操作系统要为每个进程建立一张页表 
注：页表通常存在PCB（进程控制块）中

![](/images/os-notes/3.内存管理-20250514-185026.jpg)

1. 一个进程对应一张页表 
2. 进程的每个页面对应一个页表项 
3. 每个==页表项==由 “页号” 和 “块号” 组成 
4. 页表记录进程==页面==和实际存放的==内存块==之间的==映射关系== 
5. 每个页表项的长度是相同的

#### 问题一：每个页表项占多少字节？

> [!question] 每个页表项占多少字节？
> Eg：系统物理内存大小为 4GB，页面大小为 4KB，页表项至少为多少字节？
> 
> 内存块大小 = 页面大小 = 4KB = 2<sup>12</sup>B
> 4GB的内存总共会被分为： 
> $$\frac{2^{32}}{2^{12}} = 2^{20}个内存块$$
> 内存块号的范围应该是 0~2<sup>20</sup>-1 
> 内存块号至少要用 20bit 来表示 
> 至少要用 3B 来表示块号（ 3B=24bit ）
> 由于页号是隐含的，因此每个页表项占3B，存储整个页表至少需要 3*(n+1) B

1. ==计算机中内存块的数量决定页表项中块号的数量==
2. 页表项连续存放，因此页号可以是隐含的，不占存储空间（类比数组）
3. 假设页表从内存地址为 X 处开始连续存放，页表项大小为 P
4. i 号页表项的存放地址为 $X + P \times i$
5. 页表记录的只是内存块号，而不是内存块的起始地址 
6. J 号内存块的起始地址= J $\times$ 内存块大小

#### 问题二：如何实现地址的转换

特点：虽然进程的各个页面是离散存放的，但是页面内部是连续存放的 

- 如果要访问逻辑地址A
	1. 确定逻辑地址A对应的“页号”P
	2. 找到P号页面在内存中的起始地址（需要查页表）
	3. 确定逻辑地址A的“页内偏移量”W 

逻辑地址 A 对应的物理地址= P 号页面在内存中的起始地址 + 页内偏移量 W

- 重定位寄存器：指明了进程在内存中的起始位置

#### 问题三：如何确定页号、页内偏移量？

==页号== = 逻辑地址 / 页面长度（取除法的整数部分） 
==页内偏移量== = 逻辑地址 % 页面长度（取除法的余数部分） 

1. ==逻辑地址==可以拆分为（页号，页内偏移量） 
2. 通过页号查询页表，可知页面在内存中的起始地址 
3. 页面在内存中的起始地址 + 页内偏移量 = 实际的物理地址

页面大小为 2<sup>K</sup> B，用二进制数表示逻辑地址，K 为页内偏移量，其余部分是页号

1. 在计算机内部，地址是用二进制表示的 
2. 如果页面大小刚好是 2 的整数幂，
3. 则计算机硬件可以很快速地把逻辑地址拆分成（页号，页内偏移量）

> [!Question] 题一：如何确定页号、页内偏移量？
>  Eg：在某计算机系统中，页面大小是 50 B，某进程逻辑地址空间大小为 200 B，
>  则逻辑地址 110 对应的页号、页内偏移量是多少？
>  
>  页号 = 110/50 = 2 
>  页内偏移量 = 110%50 =10 

> [!question] 题二：如何确定页号、页内偏移量？
> 假设某计算机用 32 个二进制位表示逻辑地址，页面大小为 4KB = 2<sup>12</sup>B = 4096B 
> 
> 0号页的逻辑地址范围应该是0~4095，用二进制表示应该是：
> 00000000000000000000<u>000000000000</u> ~ 00000000000000000000<u>111111111111</u> 
> 
> 1号页的逻辑地址范围应该是4096~8191，用二进制表示应该是：
> 00000000000000000001<u>000000000000</u> ~ 00000000000000000001<u>111111111111</u> 
> 
> 2号页的逻辑地址范围应该是8192~12287，用二进制表示应该是：
> 00000000000000000010<u>000000000000</u> ~ 00000000000000000010<u>111111111111</u>
> 
>Eg：逻辑地址2，用二进制表示应该是00000000000000000000<u>000000000010 </u>
>页号 = 2/4096 = 0 = 00000000000000000000
>页内偏移量 = 2%4096 = 2 = 000000000010 
>
>Eg：逻辑地址4097，用二进制表示应该是00000000000000000001<u>000000000001</u> 
>页号 = 4097/4096 = 1 = 00000000000000000001
>页内偏移量 = 4097%4096 = 1 = 000000000001

#### 问题四：为什么页面大小要取 2 的整数幂？

1. 逻辑地址的拆分更加迅速：如果每个页面大小为2KB，用二进制数表示逻辑地址，则末尾 K 位即为页内偏移量，其余部分就是页号。因此，如果让每个页面的大小为2的整数幂，计算机硬件就可以很方便地得出一个逻辑地址对应的页号和页内偏移量，而无需进行除法运算，从而提升了运行速度。
2. 物理地址的计算更加迅速：根据逻辑地址得到页号，根据页号查询页表从而找到页面存放的内存块号，将二进制表示的内存块号和页内偏移量拼接起来，就可以得到最终的物理地址。

### 基本地址变换地址

基本地址变换机构可以借助进程的页表将逻辑地址转换为物理地址

1. 通常会在系统中设置一个页表寄存器(PTR）
2. PTR 存放页表在内存中的起始地址 F 和页表长度 M
3. 当进程未执行时，页表的始址和页表长度放在进程控制块 PCB 中
4. 当进程被调度时，操作系统内核会把它们放到页表寄存器中

![](/images/os-notes/3.内存管理-20250528-110629.jpg)

- 页表项长度，页表长度，页面大小
	1. 页表长度是指这个页表中总共有几个页表项，即总共有多少页。
	2. 页面大小是指一个页面占多大的存储空间。
	3. 页表项长度是指每个页表项占多大的存储空间。

### 具有快表的地址变换机构

#### 快表（TLB）

快表，是一种访问速度比内存快很多的==高速缓冲存储器==

1. 快表又称联想寄存器（TLB，translation lookaside buffer ）
2. TLB 不是内存
3. 快表用来存放==最近访问的页表项的副本==，可以加速地址变换的速度
4. 如果 TLB 装满，选择淘汰一些页表项 —— 置换算法
5. 快表是基本地址变换机构的改进版本
6. 与此对应，内存中的页表常称为慢表

| 更贵 更快 |             |         | 更便宜 更慢 |
| ----- | ----------- | ------- | ------ |
| 寄存器   | 高速缓存（Cache） | 内存（RAM） | 外存（硬盘） |

![](/images/os-notes/3.内存管理-20250515-004639.jpg)

#### 引入快表后，地址的变换过程

1. 由 CPU 给出逻辑地址，由某个硬件算得页号、页内偏移量
2. 将页号与快表中的所有页号进行比较 
3. 如果找到匹配的页号
	1. 说明要访问的页表项在快表中有副本
	2. 则从中取出该页对应的内存块号，将内存块号与页内偏移量拼接形成物理地址
	3. 最后，==访问==该物理地址对应的==内存单元==
	4. 因此，若==快表命中==，则访问某个逻辑地址仅需==一次访存==即可 
4. 如果没有找到匹配的页号
	1. 则需要==访问内存中的页表==
	2. 找到对应页表项，得到页面存放的内存块号
	3. 再将内存块号与页内偏移量拼接形成物理地址
	4. 最后，访问该物理地址对应的内存单元
	5. 因此，若==快表未命中==，则访问某个逻辑地址需要==两次访存==

注意：在找到页表项后，应同时将其存入快表，以便后面可能的再次访问。但若快表已满，则必须按照一定的算法对旧的页表项进行替换 

1. 由于查询快表的速度比查询页表的速度快很多
2. 因此只要快表命中，就可以节省很多时间
3. 因为局部性原理，一般来说快表的命中率可以达到90%以上

![](/images/os-notes/3.内存管理-20250515-010252.jpg)

> [!question] 快表问题
> 例：某系统使用基本分页存储管理，并采用了具有快表的地址变换机构
> 访问一次快表耗时 1 us，访问一次内存耗时 100 us
> 若快表的命中率为90%，那么访问一个逻辑地址的平均耗时是多少？
>
> $$(1 + 100) \times 0.9 + (1 + 100 + 100) \times 0.1 = 111  \mu s$$
> 
> 若支持==快表和慢表同时查找==，平均耗时应该是  
> $$ (1 + 100) \times 0.9 + (100 + 100) \times 0.1 = 110.9 \mu s$$
> 
> 若未采用快表机制，则访问一个逻辑地址需要 $100 + 100 = 200 \mu s$

#### 局部性原理

- 时间局部性：
	1. 如果执行了程序中的某条指令，那么不久后这条指令很有可能被再次执行
	2. 如果某个数据被访问过，不久之后该数据很可能再次被访问
	3. 因为程序中存在大量的循环

- 空间局部性：
	1. 一旦程序访问了某个存储单元，不久后其附近的存储单元也很有可能被访问
	2. 因为很多数据在内存中都是连续存放的

在==基本地址变换机构==中，每次要访问一个逻辑地址，都需要查询内存中的页表。
由于局部性原理，可能连续很多次查到的都是同一个页表项。

### 两级页表

#### 单级页表的问题

1. 要在==所有的页表项都连续存放==的基础上才能按顺序找到页表项
2. 进程在一段时间内可能只需要访问某几个页面，没有必要让整个页表都常驻内存

- 根据页号查询页表的方法：
	1. K 号页对应的页表项存放位置 = 页表始址 + K* 4 
	2. 要在==所有的页表项都连续存放==的基础上才能用这种方法找到页表项

> [!Note] 某计算机系统按字节寻址，支持 32 位的逻辑地址，采用分页存储管理，       页面大小为 4 KB，页表项长度为 4 B，需要多少个页框来存储该页表
> 
> 4 KB = 2<sup>12</sup> B，因此页内地址要用 12 位表示，剩余 20 位表示页号 
> 因此，该系统中用户进程最多有 2<sup>20</sup> 页
> 相应的，一个进程的页表中，最多会有 2<sup>20</sup> = 1 M = 1,048,576 个页表项
> 所以一个页表最大需要 $$2^{20} \times 4\text{B} = 2^{22}\text{B}$$ 
> 共需要 2<sup>22</sup>/2<sup>12</sup>=2<sup>10</sup> 个页框存储该页表

#### 如何解决单级页表的问题？

- 使用页目录表
	1. 把页表再分页并离散存储
	2. 再建立一张页表记录页表各个部分的存放位置
	3. 或称外层页表、顶层页表

- 使用标志位
	1. 可以在需要访问页面时才把页面调入内存（虚拟存储技术）
	2. 可以在页表项中增加一个标志位，用于表示该页面是否已经调入内存
	3. 若想访问的页面不在内存中，则产生缺页中断（内中断/异常），再从外存调入内存

![](/images/os-notes/3.内存管理-20250515-110333.jpg)

#### 单级页表结构

> [!note] 单级页表结构
> 若有 32 位逻辑地址空间，页表项大小为 4 B，页面大小为 4 KB，则页内地址占 12 位
> 
> 1. 进程最多有 2<sup>20</sup> 个页面， 用 20 位二进制刚好可以表示 0~2<sup>20</sup>-1 个页号 
> 2. 每个页面可存放 4 K/4 = 1 K = 2<sup>10</sup> = 1024 个页表项

#### 地址变换的实现

- 过程
	1. 按照地址结构将逻辑地址拆分成三部分 
	2. 从 PCB 中读出页目录表始址
	3. 再根据一级页号查页目录表，找到下一级页表在内存中的存放位置 
	4. 根据二级页号查二级页表，找到最终想访问的内存块号 
	5. 结合页内偏移量得到物理地址

1. 若分为两级页表后，页表依然很长，则可以采用更多级页表
2. 一般来说各级页表的大小不能超过一个页面
3. N 级页表访问一个逻辑地址需要 N+1 次访问内存

- 两级页表的访存次数分析（假设没有快表机构） 
	1. 第一次访存：访问内存中的页目录表 
	2. 第二次访存：访问内存中的二级页表 
	3. 第三次访存：访问目标内存单元

> 最终要访问的内存块号为 4 
> 该内存块的起始地址为 $4 \times 4096 = 16384$
> 页内偏移量为 4095 
> 最终的物理地址为 16384 + 4095 = 20479

### 基本分段存储管理方式

#### 知识总览

![](/images/os-notes/3.内存管理-20250515-113741.jpg)

#### 分段

进程的地址空间：按照程序==自身的逻辑关系划分为若干个段==
每个段都有一个段名（在低级语言中，程序员使用段名来编程），每段从0开始编址 

内存分配规则：以段为单位来分配，每个段在内存中占据连续空间，各段间可以不相邻

- 由于是按逻辑功能模块划分，用户编程更方便，程序的可读性更高 

```cpp
LOAD  1,[D] | <A>;    //将分段D中A单元内的值读入寄存器1 
STORE 1,[X] | <B>;    //将寄存器1的内容存入X分段的B单元中
```

![](/images/os-notes/3.内存管理-20250515-114712.jpg)

![](/images/os-notes/3.内存管理-20250515-114947.jpg)

- 分段系统的逻辑地址结构由段号（段名）和段内地址（段内偏移量）所组成
	1. 段号的位数决定了每个进程最多可以分几个段 
	2. 段内地址位数决定了每个段的最大长度是多少

#### 段表

1. 程序分多个段，各段离散地装入内存
2. 为了保证程序能正常运行，必须从物理内存中找到各个逻辑段的存放位置
3. 为此，需为每个进程建立一张==段映射表==，简称 “段表”

- 每个段对应一个段表项，记录了该段在内存中的起始位置（基址）和段的长度 
- 各个段表项的长度是相同的

![](/images/os-notes/3.内存管理-20250516-001440.jpg)

#### 地址变换

CPU执行指令时需要将逻辑地址变换为物理地址

机器指令中的逻辑地址用二进制表示：0000000000000010<u>0000000100000000</u>

![](/images/os-notes/3.内存管理-20250516-001813.jpg)

#### 分段、分页管理的对比

- ==页是信息的物理单位==
	1. 分页的主要目的是为了实现离散分配，提高内存利用率
	2. 分页仅仅是系统管理上的需要，完全是系统行为，==对用户是不可见的==

- ==段是信息的逻辑单位==
	1. 分段的主要目的是更好地满足用户需求
	2. 一个段通常包含着一组属于一个逻辑模块的信息
	3. ==分段对用户是可见的==，用户编程时需要显式地给出段名

页的大小固定且由系统决定；段的长度却不固定，决定于用户编写的程序

1. 分页的用户进程==地址空间是一维==的，程序员只需给出一个记忆符即可表示一个地址 
2. 分段的用户进程==地址空间是二维==的，程序员在标识地址时，要给出段名和段内地址

![](/images/os-notes/3.内存管理-20250516-002419.jpg)

1. ==分段比分页更容易实现信息的共享和保护==
2. 不能被修改的代码称为纯代码或可重入代码（不属于临界资源）
3. 不能修改的代码是可以共享的
4. 可以修改的代码是不能共享的（各进程并发地访问代码段可能造成数据不一致）

- 访问一个逻辑地址需要几次访存？ 
	- 分页（单级页表）：总共两次访存 
		1. 第一次访存——查内存中的页表
		2. 第二次访存——访问目标内存单元
	- 分段：总共两次访存 
		1. 第一次访存——查内存中的段表
		2. 第二次访存——访问目标内存单元

与分页系统类似，分段系统中也可以引入快表机构
将近期访问过的段表项放到快表中，这样可以少一次访问，加快地址变换速度

### 段页式存储管理方式

#### 分页、分段的优缺点

| <img width = 70> | 优点                          | 缺点                                  |
| ---------------- | --------------------------- | ----------------------------------- |
| 分页管理             | 内存空间利用率高，不会产生外部碎片，只有少量的页内碎片 | 不方便按照逻辑模块实现信息的共享和保护                 |
| 分段管理             | 很方便按照逻辑模块实现信息的共享和保护         | 如果段长过大，为其分配很大的连续空间会很不方便。段式管理会产生外部碎片 |

段页式管理就是将分页管理和分段管理的优缺点结合起来

#### 段页式的定义

- 基本思想：
	1. 将进程按逻辑模块分段，再将各段分页（如每个页面 4 KB）
	2. 再将内存空间分为大小相同的内存块/页框/页帧/物理块
	3. 进程的各页面分别装入各内存块中

![](/images/os-notes/3.内存管理-20250528-114605.jpg)

#### 段页式管理的逻辑结构

| 31               ……            16 | 15             ……            12 | 11               ……             0 |
| :-------------------------------: | :-----------------------------: | :-------------------------------: |
|                段号                 |               页号                |               页内偏移量               |

段页式系统的逻辑地址结构由段号、页号、页内地址（页内偏移量）组成

1. 段号的位数决定了每个进程最多可以分为多少段
2. 页号的位数决定了每个段最大有多少页
3. 页内偏移量决定了页面大小、内存块大小

#### 段内寻址

![](/images/os-notes/3.内存管理-20250528-121709.jpg)

1. 每个段对应一个段表项
2. 每个段表项由段号、页表长度、页表存放块号 (页表起始地址) 组成
3. 每个段表项长度相等，段号是隐含的
4. 每个页面对应一个页表项
5. 每个页表项由页号、页面存放的内存块号组成。

#### 传统存储管理方式的特征、缺点

![](/images/os-notes/3.内存管理-20250516-090637.jpg)

==一次性==：作业必须一次性全部装入内存后才能开始运行

1. 作业很大时，不能全部装入内存，导致==大作业无法运行==
2. 当大量作业要求运行时，仅有少量作业能运行
3. 由于内存容量限制，导致==程序并发度下降==
4. 可用虚拟存储技术解决问题

驻留性：一旦作业被装入内存，就会==一直驻留在内存==中，直至作业运行结束

1. 事实上，在一个时间段内，只需要访问作业的一小部分数据即可正常运行
2. 这就导致了内存中会驻留大量的、暂时用不到的数据，浪费了宝贵的内存资源

#### 局部性原理

时间局部性：

1. 如果执行了程序中的某条指令，那么不久后这条指令很有可能再次执行
2. 如果某个数据被访问过，不久之后该数据很可能再次被访问
3. 因为程序中存在大量的循环

空间局部性：

1. 一旦程序访问了某个存储单元，不久后其附近的存储单元也很有可能被访问
2. 因为很多数据在内存中都是连续存放的，程序的指令也是顺序地在内存中存放的

| 容量小 速度快 成本高 |      |     | 容量大 速度慢 成本低 |
| :---------: | :--: | :-: | :---------: |
|     寄存器     | 高速缓存 | 内存  | 外存（如磁盘、磁带）  |

- 如何应用局部性原理？
	1. 利用高速缓冲技术的思想： 
	2. 将近期会频繁访问到的数据放到更高速的存储器中
	3. 暂时用不到的数据放在更低速存储器中

快表机构就是将近期常访问的页表项副本放到更高速的联想寄存器中

#### 虚拟内存的定义和特征

- 基于局部性原理：
	1. 在程序装入时，可以将程序中很快会用到的部分装入内存
	2. 暂时用不到的部分留在外存， 就可以让程序开始执行 

- 在程序执行过程中：
	1. 当所访问的信息不在内存时，由操作系统负责将所需信息从外存调入内存
	2. 再继续执行程序
	3. 若内存空间不够，由操作系统负责将内存中暂时用不到的信息换出到外存

==虚拟内存是通过磁盘空间扩展让程序能使用比实际物理内存更大空间的扩展技术==

- 知识点： 
	1. 虚拟内存的最大容量是由计算机的地址结构（CPU寻址范围）确定
	2. 虚拟内存的实际容量 = min（内存和外存容量之和，CPU寻址范围）
	3. 体现操作系统虚拟性，实际物理内存大小没有变，只是在逻辑上进行了扩充

> [!question] 例：计算机地址结构为32位，按字节编址，内存为512MB，外存为2GB
> 则虚拟内存的最大容量为 2<sup>32</sup>B = 4GB
> 虚拟内存的实际容量 = min(2<sup>32</sup>B , 512MB+2GB) = 2GB + 512MB

- 虚拟内存的三个主要特征： 
	1. 多次性：无需在作业运行时一次性全部装入内存，而是允许被分成多次调入内存 
	2. 对换性：在作业运行时无需一直常驻内存，允许运行过程中，将作业换入、换出
	3. 虚拟性：从逻辑上扩充了内存的容量，使用户看到的内存容量远大于实际的容量

#### 如何实现虚拟内存技术

1. 虚拟内存技术，允许一个作业分多次调入内存
2. 如果采用连续分配方式，会不方便实现
3. 因此， 虚拟内存的实现需要建立在==离散分配的内存管理方式==基础上

![](/images/os-notes/3.内存管理-20250516-094957.jpg)

- 主要区别：
	1. 在程序执行过程中，当所访问的信息不在内存时，
	2. 由操作系统负责将所需信息从外存调入内存，然后继续执行程序。
	3. 操作系统要提供请求调页（或请求调段）功能
	4. 若内存空间不够，由操作系统负责将内存中暂时用不到的信息换出到外存
	5. 操作系统要提供页面置换（或段置换）的功能

7. 若该页面在内存期间被修改过，则要将其写回外存，未修改过的页面不用写回外存

### 页面置换算法

####  知识总览

请求分页存储管理与基本分页存储管理的主要区别： 
1. 程序所访问的信息不在内存时，由操作系统负责将其从外存调入内存，再执行程序
2. 若内存空间不够，由操作系统负责将内存中暂时用不到的信息换出到外存

![](/images/os-notes/3.内存管理-20250516-150104.jpg)

用==页面置换算法==决定应该换出哪个页面
为避免较大性能开销，因此好的页面置换算法应该追求更少的缺页率

#### 最佳置换算法（OPT）

每次选择淘汰的页面将是==最长时间内或永远不再被访问的页面==，保证最低的缺页率

1. 实际上，只有在进程执行的过程中才知道接下来会访问哪个页面。
2. 操作系统无法提前预判页面访问序列。因此，最佳置换算法是无法实现的。

![](/images/os-notes/3.内存管理-20250516-150910.jpg)

#### 先进先出置换算法（FIFO）

先进先出置换算法（FIFO）：每次选择淘汰的页面是最早进入内存的页面 

- 实现方法：
	1. 把调入内存的页面根据调入的先后顺序排成一个队列
	2. 需要换出页面时选择队头页面即可
	3. 队列的最大长度取决于系统为进程分配了多少个内存块

![](/images/os-notes/3.内存管理-20250516-151028.jpg)

![](/images/os-notes/3.内存管理-20250516-151112.jpg)

==Belady 异常==——当为进程分配的物理块数增大时，缺页次数不减反增的异常现象。 

1. ==只有FIFO算法会产生Belady异常==。
2. 另外，FIFO算法虽然实现简单，但是该算法与进程实际运行时的规律不适应
3. 因为先进入的页面也有可能最经常被访问
4. 因此，算法性能差

#### 最近最久未使用置换算法（LRU）

最近最久未使用置换算法（LRU，least recently used）

最近最久未使用置换算法：每次淘汰的页面是最近最久未使用的页面 

- 实现方法：
	1. 在页表项中，用访问字段记录该页面自上次被访问以来所经历的时间 t
	2. 需要淘汰一个页面时，选择现有页面中 t 值最大的，即最近最久未使用的页面

1. 该算法的实现需要专门的硬件支持，虽然算法性能好， 但是实现困难，开销大
2. 在手动做题时，若需要淘汰页面，可以逆向检查此时在内存中的几个页面号
3. 在逆向扫描过程中最后一个出现的页号就是要淘汰的页面

![](/images/os-notes/3.内存管理-20250516-151555.jpg)

#### 时钟置换算法（CLOCK）

最佳置换算法性能最好，但无法实现；先进先出置换算法实现简单，但算法性能差；最近最久未使用置换算法性能好，是最接近 OPT 算法性能的，但是实现起来需要专门的硬件支持，算法开销大。 时钟置换算法是一种性能和开销较均衡的算法，又称CLOCK算法，或最近未用算法（NRU，Not Recently Used）

时钟置换算法（CLOCK算法），又称最近未用算法（NRU，Not Recently Used）

- 简单的CLOCK算法实现方法：
	1. 为每个页面设置一个访问位，再将页面通过链接指针链接成一个循环队列
	2. 当某页被访问时，其访问位置为1
	3. 当需要淘汰一个页面时，只需检查页的访问位。
		1. 如果是 0，就选择该页换出；
		2. 如果是 1，则将它置为 0，暂不换出，继续检查下一个页面
	4. 若第一轮扫描中所有页面都是1，将其访问位依次置为0后，再进行第二轮扫描
	5. （因此简单的CLOCK算法选择一个淘汰页面最多会经过两轮扫描）

![](/images/os-notes/3.内存管理-20250516-152251.jpg)

#### 改进型的时钟置换算法

1. 简单的时钟置换算法仅考虑到一个页面最近是否被访问过
2. 事实上，如果被淘汰的页面没有被修改过， 就不需要执行I/O操作写回外存
3. ==只有被淘汰的页面被修改过时，才需要写回外存==

因此，除考虑一个页面最近有没有被访问过外，操作系统还应考虑页面有没有被修改过

- 算法核心思想：其他条件都相同时，应优先淘汰没有修改过的页面，避免 I/O 操作

1. 设置修改位，修改位=0，表示页面没有被修改过；修改位=1，表示页面被修改过
2. ==改进型CLOCK置换算法选择一个淘汰页面最多会进行四轮扫描==

### 页面分配策略

#### 知识总览

#### 页面分配、置换策略

驻留集：指请求分页存储管理中给进程分配的物理块的集合

1. 在采用了虚拟存储技术的系统中，驻留集大小一般小于进程的总大小。 
2. 若驻留集太小，会导致缺页频繁，造成较大性能浪费
3. 若驻留集太大，又会导致多道程序并发度下降，资源利用率降低。

- 页面分配
	1. 固定分配：为每个进程分配一组固定数目的物理块，驻留集大小不变 
	2. 可变分配：先为每个进程分配一定物理块，在进程运行期间中可变

- 置换策略
	1. 局部置换：发生缺页时只能选进程自己的物理块进行置换
	2. 全局置换：可以将操作系统保留的空闲物理块分配给缺页进程，也可以将别的进程持有的物理块置换到外存，再分配给缺页进程。

![](/images/os-notes/3.内存管理-20250516-154011.jpg)

#### 固定分配局部置换

1. 系统为每个进程分配一定数量的物理块，在整个运行期间都不改变	
2. 若进程在运 行中发生缺页，则只能从该进程在内存中的页面中选出一页换出
3. 然后再调入需要的页面

- 缺点：很难在刚开始就确定应为每个进程分配多少个物理块才算合理。（采用这种策略的系统可以根据进程大小、优先级、或是根据程序员给出的参数来确定为一个进程分配的内存块数）

#### 可变分配全局置换

1. 刚开始会为每个进程分配一定数量的物理块
2. 操作系统会保持一个空闲物理块队列
3. 当某进程发生缺页时，从空闲物理块中取出一块分配给该进程
4. 若已无空闲物理块，则可选择一个==未锁定的页面==换出外存，再将该物理块分配
5. 采用这种策略时，==只要某进程发生缺页， 都将获得新的物理块==
6. 仅当空闲物理块用完时，系统才选择一个未锁定的页面调出
7. 被选择调出的页可能是系统中任何一个进程中的页
8. 因此这个==被选中的进程拥有的物理块会减少，缺页率会增加==

#### 可变分配局部置换

1. 刚开始会为每个进程分配一定数量的物理块
2. 当某进程发生缺页时，只允许从该进程自己的物理块中选出一个进行换出外存
3. 如果进程在运行中频繁地缺页，系统会为该进程多分配几个物理块
4. 直至该进程缺页率趋势适当程度
5. 反之，如果进程在运行中缺页率特别低，则可适当减少分配给该进程的物理块

可变分配全局置换：只要缺页就给分配新物理块 
可变分配局部置换：要根据发生==缺页的频率==来动态地增加或减少进程的物理块

#### 何时调入页面

预调页策略：

1. 根据局部性原理，一次调入若干个相邻的页面可能比一次调入一个页面更高效。
2. 但如果提前调入的页面中大多数都没被访问过，则又是低效的。
3. 因此可以预测不久之后可能访问到的页面，将它们预先调入内存，
4. 但目前预测成功率只有 50% 左右。
5. 故这种策略==主要用于进程的首次调入==（运行前调入），
6. 由程序员指出应该先调入哪些部分。 

请求调页策略：

1. 进程在==运行期间发现缺页时才将所缺页面调入内存==（运行时调入）。
2. 由这种策略调入的页面一定会被访问到，
3. 但由于每次只能调入一页，而每次调页都要磁盘 I/O操作，因此 I/O 开销较大。

#### 抖动（颠簸）现象

抖动（颠簸）：频繁发生刚刚换出/换入的页面马上又换入/换出内存的行为

1. 主要原因是进程频繁访问的页面数目高于可用的物理块数（分配的物理块不够）
2. 为进程分配的物理块太少，会使进程发生抖动现象
3. 为进程分配的物 理块太多，又会降低系统整体的并发度，降低某些资源的利用率

#### 工作集

为了研究应该为每个进程分配多少个物理块，Denning 提出了进程 “工作集” 的概念

- 驻留集：指请求分页存储管理中给进程分配的内存块的集合。 
- 工作集：指在某段时间间隔里，进程实际访问页面的集合。

1. 操作系统会根据“窗口尺寸”来算出工作集
2. 基于局部性原理，进程在一段时间内访问的页面与不久后访问的页面是有相关性 
3. 因此，可以根据进程近期访问的页面集合（工作集）来设计一种页面置换算法
4. 比如选择一个不在工作集中的页面进行淘汰

- 工作集大小可能小于窗口尺寸
	1. 操作系统可统计进程的工作集大小，根据工作集大小给进程分配若干内存块
	2. 如果操作系统经过一段时间的监测发现某进程的工作集最大为3
	3. 说明其有很好的局部性，给这个进程分配3个以上的内存块即可满足需要
	4. 一般驻留集大小不能小于工作集大小，否则进程运行过程中将频繁缺页
